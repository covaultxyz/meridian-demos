Summary

Lex terminal connects directly to the MERIDIAN mesh at meridian-prime rather than running a local NATS server, requiring correct NATS URL configuration and connectivity verification through a multi-phase runbook.

Decisions Made

  • Lex connects to meridian-prime (100.112.130.22:4222) not local NATS at localhost:4222
  • Do NOT install a local NATS server on Lex
  • Use NATS_AUTH_TOKEN from shared mesh credentials stored in /etc/lex/env

Key Discoveries

  • Using localhost:4222 NATS URL on Lex causes daemons to start but silently fail to register on mesh (no mesh_status() appearance, no DAEMON_CLAIMS population)
  • Wrong NATS connectivity manifests as broken daemons when the only problem is incorrect URL configuration
  • Tailscale relay routing can cause >200ms latency on NATS operations and may require direct path establishment

Lex Terminal — NATS Mesh Connectivity Runbook

**Machine**: bacherman-caselaw (100.91.70.47 Tailscale)

**NATS target**: meridian-prime (100.112.130.22:4222) — main MERIDIAN mesh

**CRITICAL**: Lex is mesh-connected. NATS is NOT local. Do NOT install a local NATS server.

---

Why This Runbook Exists

RPM Capital is a ghost terminal with isolated NATS. Lex is different: it connects directly to the

main MERIDIAN mesh. Every NATS URL in this codebase that says `nats://localhost:4222` is WRONG for

Lex — the correct URL is `nats://100.112.130.22:4222`.

Getting this wrong means daemons start but never appear on `mesh_status()`, DAEMON_CLAIMS never

populate, and session_register() silently fails. The symptoms look like daemons are broken when the

only problem is the wrong NATS URL.

Phase 1 — Tailscale Connectivity

```bash

On big-white: confirm Tailscale peer route to meridian-prime is live

tailscale ping 100.112.130.22

Expected: pong from 100.112.130.22 (... via ...) in <50ms

If timeout: check Tailscale status

tailscale status

meridian-prime should appear in peer list

```

If meridian-prime doesn't appear:

```bash

Check Tailscale is running

systemctl status tailscaled

Re-authenticate (if expired)

tailscale up

```

Phase 2 — NATS CLI Install

```bash

On big-white

NATS_VERSION=$(curl -s https://api.github.com/repos/nats-io/natscli/releases/latest | grep tag_name | cut -d'"' -f4)

wget "https://github.com/nats-io/natscli/releases/download/${NATS_VERSION}/nats-${NATS_VERSION#v}-linux-amd64.zip"

unzip nats-*.zip

mv nats-*/nats /usr/local/bin/

nats --version

```

Phase 4 — Verify Existing Streams (READ ONLY — DO NOT CREATE)

The main mesh already has all required streams. Verify they exist — do not recreate.

```bash

List all streams

nats stream ls $NATS_ARGS

Verify specific streams exist

for stream in WORK_QUEUE TASK_DISPATCH PULSE MESH_DLQ CLI_INBOX FACT_EXTRACT RESEARCH_INGEST; do
```

Expected: all 7 streams show EXISTS. If any are MISSING — check meridian-prime health, don't create on big-white.

Common Failures

Failure: `nats: no servers available`

**Cause**: Tailscale route to meridian-prime is down.

**Fix**:

```bash

tailscale ping 100.112.130.22

If no pong: tailscale up; wait 10s; retry

```

Failure: `nats: authorization violation`

**Cause**: Wrong NATS_AUTH_TOKEN in `/etc/lex/env`.

**Fix**: Retrieve the shared mesh token from the pass store (same token used by all VPS daemons).

```bash

On M1: check the current token

pass show meridian/nats-auth-token

Update /etc/lex/env on big-white with the correct value

```

Failure: `session_register` succeeds but daemon doesn't appear in mesh_status()

**Cause**: Daemon running with wrong NATS_URL (pointing to localhost:4222 which doesn't exist).

**Fix**: Check the systemd drop-in override for the daemon:

```bash

systemctl cat meridian-daemon@martin | grep NATS_URL

Should show: NATS_URL=nats://100.112.130.22:4222

If showing localhost → fix the drop-in lex.conf and restart

```

Failure: Latency >200ms on NATS operations

**Cause**: Tailscale relay routing instead of direct connection.

**Fix**:

```bash

tailscale ping --until-direct 100.112.130.22

Wait for direct path to establish (may take 30s)

```
