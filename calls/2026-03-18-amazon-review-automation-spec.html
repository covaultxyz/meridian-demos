Amazon Review Removal — Automation Specification

**Current State**: Bitler + wife run a manual review removal service using custom GPTs

**Opportunity**: Automate and scale via Meridian

**Revenue**: $175/removal (competitor charges $250)

**Strategic Value**: Quick-win product, trust-builder for larger engagements, gateway to the Amazon seller community (400-500 contacts)

---

Current Process (Manual)

1. Seller submits negative review(s) they want removed

2. Wife reviews each review against Amazon's Terms of Service

3. Custom GPT (one per violation type) generates a removal request

4. Request submitted to Amazon via Seller Central

5. If denied, iterate — sometimes 10-12 different requests per review

6. Each iteration reframes the violation argument from a different angle

7. Average success rate: unknown (but profitable enough to sustain the business)

8. Pricing: $175 per successful removal (only charge on success)

Pain Points

• **Manual, labor-intensive**: Each review can require 10-12 separate requests

• **Not scalable**: Limited by human time to review, submit, iterate

• **No tracking system**: Likely using spreadsheets or memory to track submissions

• **No analytics**: Don't know which violation types have highest success rates

• **One-at-a-time**: Can't batch process multiple sellers' reviews simultaneously

---

Automation Architecture

Phase 1: Streamline (Week 1-2)

**Goal**: Reduce manual effort per review from 30-60 minutes to 5 minutes.

| Component | What It Does |

|-----------|-------------|

| **Review intake portal** | Seller submits Amazon URL(s). System scrapes review text, star rating, product ASIN, reviewer profile, review date |

| **ToS violation classifier** | AI agent analyzes review against Amazon's ToS. Identifies specific violation type(s): irrelevant content, competitor sabotage, profanity/abuse, product confusion, policy violation, promotional content, etc. |

| **Request generator** | Specialized agent for each violation type generates the removal request. Uses the existing custom GPT logic but with MIG-backed knowledge of successful past requests |

| **Submission tracker** | Every request logged: date, violation type, angle used, Amazon response, next iteration strategy |

Phase 2: Scale (Week 3-6)

**Goal**: Handle 50+ reviews per week with minimal human intervention.

| Component | What It Does |

|-----------|-------------|

| **Batch processing** | Ingest a seller's entire review catalog, prioritize by impact (reviews closest to rounding thresholds first), process in parallel |

| **Iteration engine** | When denied, AI automatically generates the next request using a different angle. Learns from which angles work for which violation types |

| **Success pattern database** | MIG stores every request → response pair. Over time, the system learns which arguments work for which violation types and Amazon reviewers |

| **Seller dashboard** | Each seller gets a status view: reviews submitted, pending, denied (with iteration count), removed, success rate |

| **Auto-escalation** | After 5 failed attempts, flag for human review with recommendation for manual override or unusual angle |

Phase 3: Intelligence (Month 2-3)

**Goal**: Proactive review monitoring and preemptive action.

| Component | What It Does |

|-----------|-------------|

| **Review monitoring** | Continuous scraping of seller product pages. Alert when new negative reviews appear |

| **Conversion impact scoring** | Calculate the revenue impact of each negative review based on current rating, rounding threshold proximity, and product sales volume |

| **Priority ranking** | Automatically prioritize which reviews to target based on ROI (a review that moves a product from 4.24 to 4.26 = massive conversion lift) |

| **Competitor abuse detection** | Pattern recognition for coordinated negative review campaigns (common in competitive Amazon categories) |

Technical Requirements

Amazon Integration

• **Seller Central API** (SP-API) for review data, product data, and potentially submission

• **Web scraping fallback** for review content if API doesn't expose it

• **Rate limiting**: Amazon throttles API calls — need to respect limits

• **Note**: Review removal requests are submitted through Seller Central's Report Abuse flow, which may not have an API. May need browser automation (Playwright/Selenium) for submission.

Meridian Components

• **MIG brain**: Store Amazon ToS, successful request templates, violation taxonomy, outcome data

• **Daemon shell**: Review removal daemon with specialized agents for classification, generation, iteration

• **Virtual teams**: Compliance team (ToS analysis), Writer team (request generation), Analyst team (impact scoring)

• **Dashboard**: Seller-facing status dashboard (could be Telegram mini-app or web)

Data to Track

• Review text, rating, date, reviewer ID, product ASIN

• Violation type(s) identified

• Each request submitted: text, date, angle, outcome

• Success/failure rates by violation type, by angle, by product category

• Revenue impact estimates per removal
