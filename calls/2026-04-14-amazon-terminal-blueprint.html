PRISM — Metro Station Architecture

Michael presented a "subway station" architecture on the call. Each station must be built in sequence, but each feeds back into earlier stations — a loop, not a pipeline. This is the client-facing framing. Infrastructure maps to it.

Line 1: Foundation

**Station 1A: Backend Attribute Engine** (FIRST PRIORITY)

The 43K SKU horse equipment client has practically empty backend fields. Jennifer's quote: "We do not have any good processes for efficient backend field updates. It's a nightmare."

What the terminal does:

• Category ontology mapping (done once per category, then reused across all clients in that category)

• Pull category report, reconcile against Amazon's taxonomy for gaps and errors

• AI-assisted attribute population following truth hierarchy (inferred fields flagged)

• **COSMO/Rufus priority**: Jennifer identified 15 specific backend attributes critical for Amazon's COSMO engine (which feeds Rufus AI). These are the first optimization target — horse equipment category.

• Bulk upload with rejection diagnostics

• **Flat file fallback**: Certain categories can't go through API — terminal generates flat files that are manually approved and uploaded by the team. This is a technical constraint, not a limitation.

• Delivery: 100% required fields populated, zero rejections, 60-75% time reduction

MERIDIAN components: Neo4j (ontology storage + reuse), Claude CLI (attribute inference), ingestion pipeline (category report parsing), batch upload engine, flat file generator.

**Station 1B: Listing Copy Engine** (concurrent with 1A)

SEO-first copy architecture: primary keywords in title, secondary in bullets, tertiary in backend (250 chars). The intelligence is in *embedded keywords* — finding longer phrases that contain shorter target terms, so one placement covers multiple keyword targets while keeping copy coherent.

What the terminal does:

• Product sheet + brand guide ingestion

• Review/Q&A scraping for voice of customer (NOT the Amazon "Voice of Customer" metric — Michael flagged this terminology conflict)

• Keyword embedding optimization

• Per-market localization (not translation)

• Delivery: Top 50 keywords indexed, >=90% first round approval, 2 revision rounds

MERIDIAN components: Neo4j (keyword graph + embedding relationships), Claude CLI (copy generation), embedding service (semantic similarity for keyword grouping).

Station 2: Localization Engine

Country-specific keywords, buyer psychology, native proofing across 11 geographies. This is NOT translation — it's per-market optimization. A listing optimized for France won't work in Italy. The manpower to do this across all products in all markets is a nightmare no agency can staff.

MERIDIAN components: Neo4j (per-market keyword graphs, locale-specific ontologies), Claude CLI (localized copy generation), embedding service (cross-market semantic comparison).

Line 2: Intelligence

Station 3: Market Intelligence

Keyword and search intelligence dashboard. Conversion rate analysis. More sophisticated than Data Dive, Helium 10, or existing tools because it's connected to the knowledge graph — it learns, compounds, and filters noise over time.

MERIDIAN components: Neo4j (trend storage + query), ingestion pipeline (Keepa/Helium 10 exports, search term reports), dashboard (web via Cloudflare Tunnel).

Station 4: Competitive Position Tracking

Real-time competitive monitoring across marketplaces. New ASIN alerts, pricing trends, BSR shifts, Buy Box share changes.

MERIDIAN components: Neo4j (competitor graph), ingestion pipeline (scraper feeds, API exports).

Station 5: Buyer Signal Engine

Reviews, Q&A, feedback, Voice of Customer section data, cases, backend signals. Not just for own products — scrapes competitor buyer signals to understand WHY buyers choose competitors. Intelligence feeds back to Station 1B (listing copy improvement).

MERIDIAN components: Neo4j (signal graph with feedback loops to listing nodes), Claude CLI (sentiment analysis, signal extraction), contradiction detection (buyer complaint vs listing claim).

Line 3: Growth Ops & Client Output

Station 6: Product Launch Protocols

Birthing period optimization (first 30-45 days). Giveaway strategy, advertising ramp scheduling. Daily AAR (After Action Review) with ranking shift recommendations.

Station 7: Advertising Architecture

Campaign architecture (SP, SB, SD). Knowledge graph-powered optimization. Gets smarter over time — reduces noise, increases coherence, focuses on exactly the issue being investigated. The 9-doc advertising reference corpus already in repo at `docs/reference/amazon-advertising/` seeds this.

Station 8: Performance Reporting & Dashboard

Multi-client dashboard designed collaboratively as system matures. Client health overview, per-client scoping, team access.

Feedback Loops

Every station enriches earlier stations:

• Station 5 (buyer signals) → Station 1B (listing copy) — competitor intelligence improves copy

• Station 3 (market intelligence) → Station 1A (backend attributes) — keyword data reveals attribute gaps

• Station 7 (advertising) → Station 3 (market intelligence) — ad performance reveals market dynamics

• Station 2 (localization) → Station 1B (copy) — locale-specific learnings improve base copy
