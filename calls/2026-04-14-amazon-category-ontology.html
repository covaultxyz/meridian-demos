Design Principles

1. **Category-centric, not product-centric.** The ontology mapper does work once per category. Products are consumers of the ontology, not its source.

2. **Marketplace-aware.** Amazon's browse tree differs by marketplace. Categories carry marketplace scope. Attributes are universal definitions but their requirement status is per-category (which is per-marketplace).

3. **COSMO-first.** The 15 COSMO/Rufus priority attributes are first-class citizens, not afterthoughts. Each attribute knows whether it's COSMO-critical.

4. **Instance-isolated.** All nodes carry `workspace_id` for multi-tenant safety. Rankona = `"rankona"`. JV channel clients get their own workspace_id.

5. **Ontology mappings are reusable.** Category → Attribute mappings created for one client transfer to any client in the same category via shared `workspace_id = "shared"` or by cloning.

Relationships

Category Hierarchy

```

(:Category)-[:CHILD_OF]->(:Category)

```

Browse tree parent-child. No properties. A saddle category is CHILD_OF equestrian, which is CHILD_OF sports.

---

Category → Attribute Mapping

```

(:Category)-[:HAS_ATTRIBUTE {required: Boolean, cosmo_rank: Integer}]->(:Attribute)

```

The core ontology mapping. Created once per category during ontology mapping (1A-9 through 1A-11). Reused across all products in that category.

| Property | Type | Description |

|----------|------|-------------|

| `required` | Boolean | Amazon requires this attribute for the category |

| `cosmo_rank` | Integer | COSMO priority rank (1 = highest, null = not COSMO) |

| `mapped_at` | DateTime | When this mapping was established |

---

Product → Category

```

(:Product)-[:BELONGS_TO]->(:Category)

```

Product classification. A product belongs to exactly one leaf category (Amazon's constraint). No properties.

---

Product → Attribute Value

```

(:Product)-[:HAS_VALUE {value: String, confidence: Float, source: String, inferred_by: String, verified_at: DateTime, previous_value: String}]->(:Attribute)

```

The attribute value for a specific product. This is where the attribute engine writes its output.

| Property | Type | Description |

|----------|------|-------------|

| `value` | String | The attribute value (always stored as string, typed by Attribute.data_type) |

| `confidence` | Float | 0.0 - 1.0. ≥0.8 auto-uploads, <0.8 flagged for human review |

| `source` | String | "brand_sheet", "listing_scrape", "llm_inferred", "manual", "competitor_reference" |

| `inferred_by` | String | LLM model used if source = "llm_inferred" (e.g., "groq/llama3") |

| `verified_at` | DateTime | When a human confirmed the value |

| `previous_value` | String | Value before last update (for rollback) |

**Truth hierarchy** (enforced by attribute inference engine, 1A-5):

1. `brand_sheet` (confirmed from brand) — highest trust

2. `manual` (human-entered)

3. `listing_scrape` (verified from live listing)

4. `competitor_reference` (inferred from competitor data)

5. `llm_inferred` (AI-generated) — lowest trust, always flagged

Product → Marketplace

```

(:Product)-[:LISTED_IN]->(:Marketplace)

```

Product availability by marketplace. Properties: none — marketplace_id is already on the Product node for the primary listing, but a product can be cross-listed.

Product → Upload Batch

```

(:Product)-[:IN_BATCH {attributes_updated: Integer, rejected: Boolean}]->(:UploadBatch)

```

Audit trail for which products were part of which upload batch.

Category → Marketplace

(:Category)-[:AVAILABLE_IN]->(:Marketplace)

Which categories exist in which marketplaces. Amazon's browse tree differs by marketplace — a category in DE may not exist in SE.

Browse Tree Guide Reference

Amazon organizes products via the Browse Tree Guide (BTG). The schema maps to BTG as follows:

```

BTG Structure Schema Mapping

───────────── ──────────────

Root Node Category (depth=0)

└── Department Category (depth=1)

└── Category Category (depth=2)

└── Sub-category Category (depth=3, leaf)

├── Required Attribute via HAS_ATTRIBUTE {required: true}

├── Recommended Attribute via HAS_ATTRIBUTE {required: false}

└── Item Type Category.item_type_keyword

```

Key BTG behaviors the schema handles:

1. **Leaf-only classification.** Products can only BELONGS_TO leaf categories (deepest level). Internal nodes are for navigation only.

2. **Item type determines attributes.** The `item_type_keyword` on a Category is what drives which attributes are required vs optional.

3. **Marketplace variation.** DE's horse saddle category may have different required attributes than US. The Category node is per-marketplace, so the HAS_ATTRIBUTE edges capture this correctly.

4. **Flat file templates.** Amazon's flat file templates are per-item-type. The Category.item_type_keyword maps directly to the flat file template name.

Cross-Client Reusability

The schema supports the JV channel ($10K-15K/month per client) via workspace isolation:

| Data Type | Scope | Rationale |

|-----------|-------|-----------|

| Marketplace nodes | Global (shared) | Amazon marketplace IDs are universal |

| Category nodes | Per-category, cross-client | Once mapped, any client in that category reuses it |

| Attribute definitions | Global (shared) | "material_type" means the same thing everywhere |

| HAS_ATTRIBUTE edges | Per-category (shared) | Ontology mapping is the reusable IP |

| Product nodes | Per-workspace (isolated) | Client product data never crosses boundaries |

| HAS_VALUE edges | Per-workspace (isolated) | Client attribute values are proprietary |

| UploadBatch nodes | Per-workspace (isolated) | Upload history is client-specific |

Reuse flow:

1. Rankona maps horse equipment category (1A-9) → Category + Attribute + HAS_ATTRIBUTE edges created

2. JV client sells horse equipment → same Category and HAS_ATTRIBUTE edges apply

3. JV client's Products get their own workspace_id → data isolation maintained

4. Ontology investment amortizes across all clients in the same category

**Tension with exclusivity** (documented in PDR blind spots #4): Category mappings built with Rankona's expertise benefit JV clients. This is a business decision, not a schema decision. The schema supports both isolation and sharing — the business chooses which mode to use.
