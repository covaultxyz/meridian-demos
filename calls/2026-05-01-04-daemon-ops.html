Phase 1 — Create Drop-In Override Files

Drop-ins override the base `meridian-daemon@.service` template. One directory per daemon.

```bash

Create all drop-in directories

for d in las martin delphi sensei watcher sandman atlas sterling avery; do

mkdir -p /etc/systemd/system/meridian-daemon@${d}.service.d

done

Write the drop-in for each daemon — same content for all Lex daemons

for d in las martin delphi sensei watcher sandman atlas sterling avery; do

cat > /etc/systemd/system/meridian-daemon@${d}.service.d/lex.conf << 'EOF'

[Service]

EnvironmentFile=/etc/lex/env

Environment=WORKSPACE_ID=lex

Environment=MIG_NEO4J_URI=bolt://100.112.130.22:7750

Environment=CASELAW_NEO4J_URI=bolt://localhost:7687

Environment=NATS_URL=nats://100.112.130.22:4222

EOF

done

Reload systemd to pick up new files

systemctl daemon-reload

```

Note: `MCP_DAEMON_NAME=%i` is already in the base `meridian-daemon@.service` template

(added in commit db5b0efb1). Do NOT add it again in the drop-in.

Phase 2 — Enable + Start Daemons

Start order matters. LAS (write gate) must be up before other P0s write to MIG.

```bash

Enable all 9 daemons (survives reboot)

for d in las martin delphi sensei watcher sandman atlas sterling avery; do

systemctl enable meridian-daemon@$d

done

Start P0 daemons in order

systemctl start meridian-daemon@las

sleep 5 # LAS needs to claim its identity lock before others start writing

systemctl start meridian-daemon@delphi

systemctl start meridian-daemon@martin

systemctl start meridian-daemon@sensei

systemctl start meridian-daemon@watcher

sleep 5

Start P1 daemons

systemctl start meridian-daemon@sandman

systemctl start meridian-daemon@atlas

systemctl start meridian-daemon@sterling

systemctl start meridian-daemon@avery

```

Common Failure Modes

Daemon fails to start — NATS unreachable

```

Error: nats: no servers available

```

Cause: Tailscale to meridian-prime is down, or wrong NATS_URL.

Fix:

```bash

Check Tailscale

tailscale ping 100.112.130.22

If timeout: tailscale up; wait 10s; retry

Verify the drop-in has the correct NATS URL (not localhost)

systemctl cat meridian-daemon@martin | grep NATS_URL

Should show: NATS_URL=nats://100.112.130.22:4222

Restart after fix

systemctl restart meridian-daemon@martin

```

Daemon fails to start — Caselaw Neo4j unavailable (Delphi, Martin)

```

Error: ServiceUnavailable: Connection refused to localhost:7687

```

Fix:

```bash

systemctl status neo4j

journalctl -u neo4j -n 50

systemctl restart neo4j

sleep 20 # Wait full 20s — 11M node graph takes time

systemctl restart meridian-daemon@delphi meridian-daemon@martin

```

Daemon starts but doesn't appear in mesh_status()

Cause: WORKSPACE_ID or NATS_URL wrong in drop-in.

Fix:

```bash

Print effective environment for the daemon

systemctl cat meridian-daemon@martin

Check WORKSPACE_ID specifically

source /etc/lex/env

systemctl show meridian-daemon@martin -p Environment

Confirm it says:

WORKSPACE_ID=lex

NATS_URL=nats://100.112.130.22:4222

MIG_NEO4J_URI=bolt://100.112.130.22:7750

If wrong: edit /etc/systemd/system/meridian-daemon@martin.service.d/lex.conf

Then: systemctl daemon-reload && systemctl restart meridian-daemon@martin

```

Zombie DAEMON_CLAIMS blocking restart

Rare: a daemon crashed mid-heartbeat and left a stale claim. The claim expires in 30s automatically,

but if you need to force-clear:

```bash

source /etc/lex/env

Clear claim on the MAIN MESH KV (not local)

nats kv del DAEMON_CLAIMS martin \

--server "nats://${NATS_AUTH_TOKEN}@100.112.130.22:4222"

systemctl start meridian-daemon@martin

```

Watcher fires P0 alert: daemon OFFLINE

Watcher's 15-minute alert cooldown (per daemon key) prevents spam. Response SLA:

• P0 daemon (las, martin, delphi, sensei, watcher): respond within 15 minutes

• P1 daemon (sandman, atlas, sterling, avery): respond within 1 hour

Standard response:

```bash

journalctl -u meridian-daemon@<daemon> -n 50

systemctl restart meridian-daemon@<daemon>

If restart fails 3x → 3-strike rule: escalate to Sensei or operator

```

Watcher fires: caselaw Neo4j OFFLINE

Watcher monitors TWO Neo4j instances. Separate alert keys:

• `neo4j:localhost:7687` — caselaw (local)

• `neo4j:100.112.130.22:7750` — main MIG (remote)

For caselaw Neo4j offline:

```bash

systemctl status neo4j

journalctl -u neo4j -n 50

Check disk space — 11M node graph can fill /data unexpectedly

df -h /data

systemctl restart neo4j

sleep 20

systemctl restart meridian-daemon@delphi meridian-daemon@martin

```
