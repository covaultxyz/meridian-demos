IP Protection Playbook

**Origin**: Bitler raised this on the March 18 call — "What keeps the user from obtaining all the knowledge that Meridian has?"

**Purpose**: Framework for protecting Meridian's IP across all deployment models

---

The Concern

Bitler identified the core vulnerability: "Competition could essentially just buy a contract, get the VPS, get all the code, spin it up, be better at selling than us."

This is a real risk. Here's how we mitigate it.

---

Meridian's IP Layers

| Layer | What It Is | Defensibility | Risk Level |

|-------|-----------|---------------|-----------|

| **1. MIG Brain** | 7,000+ memory nodes, 10,000+ concepts, curated ontology, years of research | **HIGH** — Can't be downloaded via API. Takes years to replicate. | Low (API-only access) |

| **2. Ontology** | Classification taxonomy across 6+ domains with weighted relationships | **HIGH** — Invisible to clients. Only manifests as accuracy. | Low |

| **3. Architecture** | Daemon orchestration, MIG pipeline, NATS messaging, knowledge graph engine | **MEDIUM** — Code is visible if client has VPS access | Moderate |

| **4. Deployment Config** | Systemd services, launchd agents, deployment scripts, health checks | **LOW** — Operational knowledge, replicable with effort | Low (not valuable alone) |

| **5. Client Configurations** | Custom workflows, domain-specific agents, industry protocols | **CLIENT-OWNED** — This is their IP, not ours | N/A |

Protection by Deployment Model

Model A: Managed VPS (Recommended Default)

**How it works**: Meridian provisions and manages the VPS. Client accesses via Telegram + dashboard only.

| Surface | Exposure | Mitigation |

|---------|----------|-----------|

| Telegram | Commands and responses only | No code exposure |

| Dashboard | Rendered KPIs and visualizations | No raw data access |

| Voice AI | Phone interface | No code exposure |

| VPS itself | Root access controlled by Meridian | Client has no SSH access |

| GitHub repo | Not cloned to client's systems | Client never sees the repo |

| MIG brain | API queries return results, not the graph | No bulk export capability |

**Risk**: Very low. Client would need to compromise the VPS to access code.

Model B: On-Premises (Client Hardware)

**How it works**: Code deployed to client's Mac Mini/Studio. Client has physical access.

| Surface | Exposure | Mitigation |

|---------|----------|-----------|

| Source code | Visible on disk if they look | Obfuscation possible but not bulletproof |

| Configuration | Visible | Expected — this is their system |

| MIG brain (local) | Accessible via Neo4j | Contains their data, not Meridian's global brain |

| Architecture | Fully exposed | Risk acknowledged |

**Mitigations**:

1. **Separate the brains**: Local MIG contains client data only. Meridian's global research and ontology are served via API from Meridian's infrastructure. Client's local brain is valuable to them but not to a competitor.

2. **License agreement**: MSA explicitly states platform architecture is Meridian IP. Code on their machine is licensed, not owned.

3. **Version control**: They get a deployed snapshot, not git access. No commit history, no branch structure, no development tooling.

4. **API-gated features**: KaaS, global research, cross-client intelligence — all served from Meridian's cloud. Even with local code, they can't replicate these.

5. **Rapid innovation**: We ship improvements monthly. A static snapshot depreciates fast.

**Risk**: Moderate. A technical buyer could study the code. But:

• They'd have the skeleton, not the brain content

• They'd have one snapshot, not the evolving platform

• They'd still lack the ontology, research pipeline, and operational knowledge

• Building from our code is still harder than building from scratch (our patterns, not theirs)

Model C: Hybrid (Client Hardware + Cloud)

**How it works**: Local compute for processing, cloud for brain backup and KaaS.

| Surface | Exposure | Mitigation |

|---------|----------|-----------|

| Local code | Visible (same as Model B) | Same mitigations as Model B |

| Cloud brain | API-only access | Same as Model A |

| KaaS features | API-only | Same as Model A |

**Risk**: Low-moderate. Best balance of performance and protection.

Contractual Protections

Already built into the MSA template:

| Protection | MSA Section | Effect |

|-----------|-------------|--------|

| **Platform IP ownership** | 4.2 | "The Platform and all Meridian Platform IP are and will remain the exclusive property of Meridian" |

| **Non-transferable license** | 4.2 | Client gets a license to use, not own, the platform |

| **Confidentiality** | 5.1 | Mutual confidentiality obligations on all non-public information |

| **Survival** | 5.4 | Confidentiality survives termination for {{YEARS}} years |

| **Non-compete (vertical)** | 4.7 | We don't build the same thing for their direct competitor |

Additions to Consider

1. **Non-reverse-engineering clause**: "Client shall not decompile, reverse engineer, disassemble, or otherwise attempt to derive the source code or architecture of the Platform."

2. **Code access restriction**: For on-prem deployments, restrict access to the application layer. Root/admin access to the Meridian services directory is not included.

3. **Audit right**: Meridian retains the right to audit on-prem deployments to verify compliance with license terms.

What to Tell Prospects Who Ask

**Simple answer**: "Your data stays yours. Our platform stays ours. You interact through Telegram and your dashboard — you never see the code. The brain that powers everything is served from our infrastructure. Think of it like Bloomberg — you use the terminal, you don't own the data center."

**For technical prospects**: "The platform runs on your hardware or our VPS, depending on your preference. Either way, the platform IP is licensed to you under our MSA. The global knowledge bases, research index, and ontology are served via API from our cloud. Your custom configurations and business data are 100% yours. If you ever leave, we export everything that's yours within 15 business days."

**If they push on the moat**: "Even if someone replicated our code line for line, they'd have an empty vessel. The value is in the brain — 10,000+ curated memories, a multi-domain ontology, years of research — and the team that maintains it. That's what you're subscribing to."
