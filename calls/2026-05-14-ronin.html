Pipeline Stages

Stage 1 — Intake

• Receive document (PDF, DOCX, HTML, or plain text)

• Verify source provenance: author, publication, date, citation/DOI if available

• Check for privileged content indicators → reject, flag to Martin if uncertain

• Hash content for dedup check before processing

Stage 2 — Chunk

• Split by natural legal structure: sections, headings, numbered paragraphs

• Legal documents are hierarchically structured — respect the hierarchy in chunks

• Target chunk size: 400-600 tokens (legal text is dense; smaller chunks lose context)

• Preserve citation context: keep case citations in the chunk that references them

Stage 3 — Embed

• Embed via Ollama on GTX 1070 Ti (all-minilm-l6-v2, 768d)

• Legal documents are confidential — NEVER send to external embedding APIs

• Batch embedding (GTX 1070 Ti can process ~500 chunks/hour)

• Store vectors alongside content in ResearchMemory nodes

Stage 4 — Classify

• Run classification against legal domain centroids

• Return domain(s), confidence scores, and rationale

• If confidence < 0.70 on top domain: flag for Vance or Martin review

• Single-domain for precision, not multi-label (except cross-domain documents)

Stage 5 — Store (via LAS)

• Package as ResearchMemory node with full provenance

• Required fields: source, author, date, domain, workspace_id=lex, chunk_index, total_chunks

• Route through LAS write gate — no direct Neo4j writes

• Write ingestion confirmation memory: `[INGEST][domain] document_title: chunks_stored, domain_confidence`

Vance Ingestion Queue

Process `[VANCE-INGEST-REQUEST]` memories from Vance:

1. Pull from mig_search: `[VANCE-INGEST-REQUEST]`
