OVERALL ASSESSMENT: Solid framework, but several provisions need tightening before signing

The agreement does a decent job establishing the IP boundary between MBM work and the Partnership (Mazzei/Bitler). The attorney clearly understood the dual-hat situation. However,

RED FLAGS (address before signing)

1. Section 2.5 — Irrevocable Power of Attorney (p.5)

▎ "Contractor hereby irrevocably appoints MBM as Contractor's attorney-in-fact to execute such documents on Contractor's behalf if Contractor fails or refuses to do so within ten

(10) business days"

Problem: This gives MBM the power to sign IP assignment documents in your name if you don't respond within 10 business days. The word "irrevocable" is the issue — even after

termination, MBM retains this power. If there's ever a dispute about whether something is Company IP vs. Pre-Existing IP, MBM can unilaterally sign away the disputed IP on your

behalf.

Recommendation: Make this power revocable upon termination. Add: "This power of attorney shall terminate upon expiration or termination of this Agreement." Also extend the response

window to 20-30 business days.

2. Section 4.3 — Right of First Refusal (p.8)

▎ "If Contractor or the Partnership develops or contemplates developing any product, service, or technology in the medical or healthcare administration software domain..."

Problem: "Healthcare administration software domain" is extremely broad. MERIDIAN/AgentVox's multi-agentic PaaS could serve any industry, but the moment it touches healthcare admin —

even as one of many verticals — you'd owe MBM a right of first refusal. This effectively gives MBM a veto on the Partnership entering the healthcare market independently.

Recommendation: Narrow the scope. Replace "medical or healthcare administration software domain" with something like "products that directly compete with the specific functionality

of the Project Deliverables (as defined in the applicable Statement of Work)". The current language captures too much of the addressable market.

3. Section 1.1 — Company IP Definition (p.3)

▎ Clause (c): "created using MBM's resources, data, or specifications"

Problem: "Resources" is undefined and dangerously broad. If MBM provides a laptop, cloud credits, or even office space, anything you create on those resources — even personal

projects during off hours — could be claimed as Company IP under this clause.

Recommendation: Define "resources" narrowly or add: "For purposes of clause (c), 'resources' means MBM's proprietary software, proprietary data sets, and documented technical

specifications, and does not include general-purpose computing equipment, cloud services, or development tools."

4. Exhibit A — Pre-Existing IP Inventory (p.17)

Problem: Only 2 items listed, one is a placeholder. This is dangerously thin. The agreement says (Sec 7.1(e)) that you represent you've disclosed all Pre-Existing IP. Anything not in

Exhibit A is vulnerable to being claimed as Company IP if MBM argues it was "created in connection with the Services."

MERIDIAN itself is not listed. The MIG, NATS mesh, Neo4j memory graph, daemon architecture, pulse system, all the orchestrator/brain/mesh code — none of this is inventoried. If any

of these tools touch MBM work (and they will, per Section 3.4), the customization layer becomes Company IP while the base tools remain yours — but only if they're in Exhibit A.

Recommendation: Before signing, add a comprehensive Exhibit A including at minimum:

• MERIDIAN distributed daemon system (architecture, orchestrator, brain, mesh)

• MIG (Memory Intelligence Graph) — Neo4j knowledge system

• AgentVox / multi-agentic PaaS platform

• All general-purpose frameworks, libraries, deployment scripts

• Voice/meeting intelligence pipeline

• Any reusable components you've built

YELLOW FLAGS (negotiate if possible)

Issue: You can't even reference your own work in a portfolio or use knowledge gained to improve your general skills applied to Company IP specifically. This is standard but

Issue: Complete waiver of attribution and integrity rights. Puerto Rico follows US copyright law where moral rights are limited anyway, but this is a total surrender. Reasonable in a

CATEGORY A: MUST FIX — Issues That Create Legal Exposure

A1. Exhibit A Is Dangerously Thin — MERIDIAN Is Not Listed

The entire Pre-Existing IP inventory has 2 entries and a placeholder. But Alex brings a production-grade distributed system to this engagement:

• MERIDIAN daemon mesh (orchestrator, brain, mesh, collaboration layers)

• MIG (Memory Intelligence Graph) — Neo4j knowledge system, 17,900+ nodes

• Voice intelligence pipeline (diart + pyannote + mlx-whisper)

• Pulse sentinel system (49 registered sentinels)

• NATS messaging infrastructure

• Blue/green deployment system

• All brain/, mesh/, orchestrator/, skills/ code

Why this matters: Section 1.1 defines Company IP as anything "conceived in connection with the Services" or "created using MBM's resources, data, or specifications." If any MERIDIAN

component touches MBM work — and it will, because the Clinical Platform will run on this infrastructure — the customization layer becomes Company IP. That's correct. But without

explicit Exhibit A listing, MBM could argue the base platform itself was created "in connection with" their project.

Omar's own position supports this fix: On March 10, the hybrid layer model was agreed — infrastructure = MERIDIAN (licensed), clinical application = Omar (owned). Exhibit A needs to

codify that exact boundary.

Resolution: Populate Exhibit A with a comprehensive inventory before signing. Minimum 15-20 line items covering every reusable system Alex brings to the engagement.

A2. Section 2.5 — Power of Attorney Survives Termination

The irrevocable attorney-in-fact clause lets MBM sign IP assignment documents in Alex's name after 10 business days of non-response. "Irrevocable" means this power survives

termination, bankruptcy, and dispute.

Scenario: Alex and Omar disagree about whether a component is Company IP or Pre-Existing IP. Agreement terminates under Section 9.3 for cause. MBM still holds the power to sign

assignment documents on Alex's behalf for the disputed IP — while the dispute is active.

Resolution: Add: "This power of attorney shall terminate upon expiration or termination of this Agreement and shall not apply to any IP that is the subject of an active dispute

between the Parties." Extend the response window to 30 business days.

A3. Section 4.3 — ROFR Scope Captures the Entire Healthcare Market

"Medical or healthcare administration software domain" is the problem phrase. AgentVox is a multi-agentic PaaS — industry-agnostic by design. But the moment Alex or the Partnership

builds a healthcare vertical (for any client, not just competing with MBM), this clause triggers a right of first refusal.

Cross-reference with March 10 call: Omar explicitly stated he wants full ownership of clinical software. Alex explicitly stated he wouldn't compete in medical. This ROFR was likely

intended to formalize that verbal agreement. But the written scope far exceeds the spoken intent.

Example of overreach: If the Partnership builds a general scheduling API and a dental practice uses it — that's "healthcare administration software" and triggers ROFR.

Resolution: Narrow to: "products or services that are substantially similar to and directly competitive with the specific Project Deliverables described in the applicable Statement

of Work." This captures the intent without blocking the Partnership's entire addressable market.

A4. Section 1.1(c) — "Resources" Is Undefined

"Created using MBM's resources, data, or specifications" — "resources" has no definition boundary. If Omar provides cloud credits, a VM, an API key, or even a Slack workspace,

anything created using those could be captured.

Cross-reference with March 16 Ely-Alex sync: The crypto mining attack on Omar's Google Cloud infrastructure was discussed. If Alex does security hardening on that infrastructure and

develops reusable security tooling, is that Company IP because it was created "using MBM's resources"? Under the current language, arguably yes.

Resolution: Define "resources" in Article 1: "'MBM Resources' means MBM's proprietary software, proprietary data sets, patient data, clinical protocols, and documented technical

specifications specific to the Project, and does not include general-purpose computing infrastructure, cloud services, development tools, or communication platforms."

A5. BAA Does Not Exist — HIPAA Gate Is Open

Exhibit B is a placeholder: "To be attached separately." But Project 2 is a "comprehensive, LAN-based clinical and administrative psychiatric platform." This will involve PHI.

Without a BAA:

• Any PHI Alex encounters during development has no governing agreement

• Alex has no safe harbor for incidental exposure

• MBM has no compliant chain of custody

• A single patient complaint could trigger HHS investigation with no contractual defense

Cross-reference with March 10 call: "BAA must be signed before go-live" was listed as a hard gate. But the IP agreement can be signed before the BAA is ready — it just means Phase 2

work cannot begin until Exhibit B is attached.

Resolution: Add a clause: "No Services related to Project 2 shall commence until Exhibit B (BAA) has been executed by both Parties." This lets the IP agreement proceed while gating

PHI exposure.

CATEGORY B: SHOULD UPDATE — Gaps That Strengthen the Agreement

B1. Compensation Schedule Is Blank

Exhibit C has no rates. March 10 context: Phase 1 was $50/hr. Phase 2 is HIPAA-compliant clinical AI — 2-3x complexity.

The agreement can't be signed with blank compensation. But more importantly: without a rate locked in, MBM controls timing. They can delay rate agreement while expecting Phase 2 work

to continue under Phase 1 terms.

Resolution: Fill Exhibit C before signing. Consider: base rate for general dev, premium rate for HIPAA/clinical work, separate line for architecture/consulting.

B2. No Statement of Work Referenced

The agreement repeatedly references "applicable Statements of Work" (Sections 1.4, 6.5) but none are attached or referenced. Project 1 and Project 2 have different scopes, risk

profiles, and compensation implications.

Resolution: Attach or reference SOWs for both projects. At minimum, add an Exhibit F with scope boundaries for Project 1 (website automation) and Project 2 (clinical platform).

B3. Exhibit E — Transfer Pricing Is a Dormant Unilateral Trigger

MBM can activate Exhibit E "at MBM's sole discretion, at any time during the term." Once activated, it governs the methodology for all intercompany transactions and has "the same

binding force as all other Exhibits."

The problem: Alex must "cooperate in its implementation" but has no input on the methodology. Transfer pricing is complex under Act 60 — the IRS specifically scrutinizes PR-based

intercompany arrangements. If MBM activates this and imposes transfer pricing terms that characterize Alex's income unfavorably, Alex is bound.

Resolution: Change from unilateral to mutual activation: "Exhibit E shall be activated upon mutual written agreement of both Parties." Or at minimum: "Contractor shall have thirty

(30) days to review and provide input on the Transfer Pricing Study methodology before it becomes binding."

B4. Section 3.5 — Segregation Obligation Falls Entirely on Contractor

Alex must "implement and maintain commercially reasonable practices to segregate work performed for MBM from work performed for Contractor's own business." This is operationally

correct — but the agreement creates no corresponding obligation for MBM to facilitate segregation (separate repos, dedicated credentials, isolated environments).

Cross-reference: March 16 Ely-Alex sync discussed the crypto mining attack on Omar's GCP. If MBM's infrastructure is compromised and Alex's tools are deployed there, the commingling

creates mutual exposure but only Alex bears the segregation obligation.

Resolution: Make bilateral: "Both Parties shall implement and maintain commercially reasonable practices to segregate..."

B5. No IP Dispute Resolution Mechanism

When a specific component is disputed — is it Company IP, Pre-Existing IP, or General Knowledge? — the agreement has no process. Section 10.1 has general dispute resolution

(mediation then arbitration), but IP classification disputes need something faster than formal arbitration.

Resolution: Add a Section 2.7: "In the event of a dispute regarding the classification of specific intellectual property, the Parties shall first attempt to resolve the dispute

through good faith discussion within fifteen (15) business days. If unresolved, either Party may engage an independent IP attorney, mutually agreed upon, whose classification

recommendation shall be binding."

B6. Section 6.4 — Subcontractor Approval Has No Timeline

MBM must provide "prior written approval" before any Partnership member touches MBM code. But no timeline for approval/denial is specified. MBM could delay indefinitely, blocking

Alex from using his own team.

Resolution: Add: "MBM shall respond to subcontractor approval requests within ten (10) business days. Failure to respond shall be deemed approval."

CATEGORY C: STRATEGIC TENSIONS — Not Errors, But Require Eyes-Open Signing

C1. The Hybrid Layer Model Is Implicit, Not Explicit

The March 10 verbal agreement was:

┌──────────────────────┬─────────────────────┐

│ Layer │ Owner │

├──────────────────────┼─────────────────────┤

│ Infrastructure │ MERIDIAN (licensed) │

├──────────────────────┼─────────────────────┤

│ Clinical Application │ Omar (owned) │

├──────────────────────┼─────────────────────┤

│ Clinical Data & PHI │ Omar (owned) │

├──────────────────────┼─────────────────────┤

│ Domain Knowledge │ Omar (owned) │

└──────────────────────┴─────────────────────┘

The agreement implicitly supports this through Sections 3.1 (Pre-Existing IP preserved), 3.2 (General Knowledge carve-out), and 3.4 (Independent Contractor Tools). But it never

explicitly states this layered model. The absence of an explicit statement means interpretation disputes will be resolved through the general IP assignment provisions of Article 2 —

which favor MBM.

Recommendation: Add a preamble to Article 3 or a new Section 3.6 that codifies the layer model: "The Parties acknowledge that the Project architecture consists of distinct layers:

infrastructure (Contractor Pre-Existing IP, licensed per Section 3.4), application (Company IP), and data (Company property). This layered model shall govern the classification of IP

components."

C2. Omar's "Full Ownership" vs. Alex's "Won't Build Outside My Platform"

On March 10, Omar said: full independent ownership, no third-party dependencies, ability to sell the practice with software included. Alex said: won't build outside MERIDIAN

platform.

The agreement resolves this through the Section 3.4 perpetual license — MBM gets a perpetual license to use embedded tools. But "perpetual non-exclusive license" is not "full

ownership." If Omar sells the practice, the buyer gets the license but not the platform source code. If Alex's infrastructure evolves and MBM's version diverges, MBM is stuck on the

embedded snapshot.

Omar needs to understand: He owns the clinical application layer and all customizations. He does not own the MERIDIAN infrastructure layer — he has a license. This distinction

matters if he sells or if the relationship ends.

C3. Act 60 Residency Certification Creates Ongoing Obligation

Section 7.4: Alex represents he's a bona fide PR resident and must notify MBM of any residency change that could affect income-sourcing. Section 7.5: Alex must cooperate with MBM's

compliance requirements including documenting "work location and hours."

This is appropriate for Act 60 structuring, but it creates an ongoing administrative burden and surveillance dynamic in what's supposed to be an independent contractor relationship.

The IRS specifically looks for control indicators that reclassify contractors as employees — documenting work location and hours is one of them.

Not a dealbreaker, but worth noting: the very compliance mechanisms that protect Act 60 status could undermine independent contractor classification if implemented too aggressively.

C4. MBM Has No Obligation to Actually Use the Deliverables

MBM owns all Company IP absolutely. MBM has no obligation to deploy, maintain, or pay ongoing fees for the infrastructure license. If the relationship sours, MBM can shelve

everything Alex built — including the clinical platform Omar wants — and Alex can't reuse any of it.

This is standard in work-for-hire agreements but worth understanding given the long-term partnership intent expressed on March 10.

CATEGORY D: UNMITIGATED RISKS — Not in the Agreement

D1. No Source Code Escrow (Cuts Both Ways)

If Alex becomes unavailable (incapacity, dispute, Act 60 residency change), MBM has a perpetual license to tools embedded in Deliverables but no access to the MERIDIAN source code

those tools depend on. The license without source access is operationally useless for maintenance.

Conversely, if MBM terminates and Alex needs to demonstrate that the base platform predates the engagement, source control history is the only evidence. No escrow means no neutral

third-party timestamp.

Exhibit D (Best Practices) mentions escrow as a recommendation but it's not in the binding agreement.

D2. The Google Cloud Crypto Mining Attack Is Unresolved

March 16 sync revealed a 6-month persistent compromise on Omar's GCP infrastructure — 30 security audit iterations, 8 firewalls, still penetrating. Any code or tools deployed to this

environment could be:

• Compromised or exfiltrated

• Used in claims about "created using MBM's resources" if the attacker used Alex's tools

• A HIPAA breach vector the moment PHI enters Phase 2

The agreement has no security standards clause. Neither party is obligated to maintain minimum security posture. Add a Section 7.6: "MBM shall maintain commercially reasonable

security measures for all infrastructure on which Deliverables are deployed."

D3. No Insurance Requirements

Neither party is required to carry professional liability, cyber liability, or E&O insurance. Given the HIPAA exposure in Phase 2, both parties should carry minimum coverage. A

single PHI breach could exceed the financial capacity of both parties.
