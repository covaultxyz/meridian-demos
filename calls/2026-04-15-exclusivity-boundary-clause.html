Purpose

This document defines the intellectual property boundary between what MERIDIAN provides exclusively to Rankona Mazon and what MERIDIAN may deploy for other clients through the JV channel. It is intended for inclusion in the Master JV Partnership Agreement and for review by Rankona shareholders (Sander and others) prior to contract signature.

**Core principle:** Rankona's investment in building this terminal creates proprietary assets that belong to Rankona. MERIDIAN's platform capabilities — the code, architecture, and methodology — remain MERIDIAN's to deploy elsewhere, but never using Rankona's data, mappings, or competitive intelligence.

---

Section 1 — Category-Exclusive IP (Rankona-Owned)

The following assets are developed with, for, and from Rankona's operations. They are **exclusive to Rankona** and will not be shared with, licensed to, or replicated for any other client — including JV channel clients.

1.1 Category Ontology Mappings

All Amazon product category ontology mappings developed using Rankona's category reports, Jennifer's domain expertise, or Rankona's SKU catalog. This includes:

• Attribute schemas for every Amazon category Rankona operates in

• COSMO/Rufus attribute prioritization mappings derived from Rankona's data

• Category-to-attribute inference rules trained on Rankona's product mix

• Any ontology updates or refinements made during the engagement

1.2 Competitive Intelligence

All competitive intelligence derived from Rankona's operations:

• Buyer signal patterns extracted from Rankona's product reviews and Q&A data

• Keyword performance data specific to Rankona's active ASINs and categories

• Ranking and suppression event history for Rankona's listings

• Market positioning analysis conducted on Rankona's behalf

1.3 Knowledge Graph Content

The accumulated knowledge graph built in Rankona's isolated Neo4j instance:

• All Memory, Concept, and ResearchMemory nodes created from Rankona's operations

• All relationships and edges in Rankona's workspace

• Historical performance data and extracted insights

• Jennifer's documented workflows, SOPs, and process knowledge captured in the system

1.4 Exit Valuation Assets

Upon any acquisition event:

• Rankona's category ontology and knowledge graph transfer to the acquirer as Rankona proprietary assets

• MERIDIAN retains no ownership claim over knowledge content generated from Rankona's operations

• Acquirer (e.g., Humble Group) receives clean title to all Rankona-specific IP listed in Sections 1.1–1.3

---

Section 2 — Platform IP (MERIDIAN-Owned, Available to JV Channel)

The following are MERIDIAN's proprietary platform assets. They are **not exclusive** to Rankona. MERIDIAN may deploy them for JV channel clients, provided no Rankona data, mappings, or competitive intelligence is used.

2.1 Platform Codebase

• All source code for the terminal platform (fact extraction, mutation pipeline, embedding service, Telegram bot framework, bulk upload engine, sentinel workers)

• Dashboard framework, query interfaces, and alert systems

• Infrastructure tooling (Cloudflare Tunnel config, Neo4j deployment, NATS mesh)

2.2 Methodology and Architecture

• The terminal station architecture (Station 1A through Station 8)

• The data contract schema between stations

• LLM offload routing and inference patterns

• The knowledge graph schema (node types, relationship types, property definitions)

2.3 General AI Capabilities

• Extraction and inference methodology (not the trained mappings)

• Keyword optimization logic (not the keyword data)

• Copy generation prompts (not Rankona-specific prompt tuning)

• Competitive monitoring architecture (not Rankona's monitored competitor list)

Section 3 — Category Exclusivity Period

To protect Rankona's competitive position, MERIDIAN agrees to the following:

**3.1 Same-Category Restriction:** For any Amazon product category in which Rankona actively operates at the time of JV channel deployment, MERIDIAN will not deploy the terminal for a direct competitor in that same category without Rankona's written consent.

**3.2 Definition of "direct competitor":** A seller whose primary product catalog overlaps with Rankona's active categories by more than 30%.

**3.3 Duration:** This restriction applies for the longer of (a) 24 months from the date of first JV channel deployment, or (b) until Rankona's exit event.

**3.4 JV channel governance:** Carl Helgesson holds veto rights over any JV channel prospect in Rankona's active categories. Veto must be exercised within 14 days of written notice from MERIDIAN.

---

Section 4 — Instance Isolation Guarantee

Each JV channel client operates in a fully isolated environment:

• **Separate Neo4j database** — JV client data never touches Rankona's workspace

• **Separate NATS namespace** — no message routing between instances

• **Separate Telegram bot** — no shared access or visibility

• **No knowledge transfer** — JV client's terminal starts from zero; Rankona's mappings are not seeded, referenced, or used as training data for JV deployments

• **Separate Cloudflare Tunnel** — JV clients access their own subdomain with no shared infrastructure

MERIDIAN maintains this isolation as a technical guarantee, not merely a policy commitment. Rankona has the right to audit instance isolation at any time.

Section 6 — Acquirer Due Diligence (Humble Group Scenario)

To the question "Is this terminal exclusive to Rankona?" — the accurate answer is:

> "The knowledge, mappings, and competitive intelligence built inside the terminal are exclusively Rankona's property. The platform architecture and code are MERIDIAN's, deployable for other clients but never using Rankona's data. Rankona retains full ownership of the IP that drives exit valuation."

This framing is clean for due diligence. The terminal's strategic value is in the accumulated knowledge, not the software — and that knowledge is exclusively Rankona's.
