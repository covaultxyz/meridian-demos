Summary

Technical runbook for wiring the MERIDIAN intelligence stack into the Rankona Mazon ghost terminal across a 2-node cluster (core + edge VPS).

Decisions Made

  • 6 daemons split 3/3 across core (forge, atlas, sensei) and edge (watcher, sandman, delphi) nodes.
  • NATS hub lives on the edge VPS; core daemons connect via Tailscale.
  • Edge daemons use claude-haiku for light ops; core daemons use claude-sonnet for high-judgment tasks.
  • Snapshot archiver enforces 90-day rolling retention on high-frequency Amazon graph nodes.
  • Evelyn client support integration deferred to Phase 2 and runs on MERIDIAN mesh, not Rankona nodes.

Key Discoveries

  • At target scale the graph reaches ~91M nodes, with KeywordRankSnapshot alone comprising ~50M nodes, requiring aggressive retention management.
  • Tailscale disconnection causes core daemons to lose NATS entirely, necessitating a local-only fallback mode.
  • Evelyn on MERIDIAN will create tickets in Rankona's Neo4j via Tailscale cross-mesh bridge without exposing the full Rankona graph.

Rankona Mazon Terminal — MERIDIAN Plumbing Runbook

**Purpose**: Wire the full MERIDIAN intelligence stack into the Rankona Mazon ghost terminal. Covers everything between "Neo4j + NATS are installed on both nodes" and "daemons are functional with all bells and whistles."

**Prerequisite**: Core machine (100.97.248.107) provisioned with Neo4j, embedding service, Claude CLI. Edge VPS provisioned with NATS, Cloudflare Tunnel, Caddy. SSH access confirmed on both nodes. Tailscale mesh between nodes operational.

**Topology**: 2-node cluster — core (100.97.248.107 via Tailscale) + edge (VPS, public-facing)

**Companion docs**: `amazon-terminal-blueprint.md`, `vps-architecture-blueprint.md`, `hardware-sizing-75k-skus.md`

Proposed Daemon Roster

The blueprint defines 8 metro stations but no daemon roster. This runbook proposes 6 daemons split 3/3 across the core/edge topology:

Core (rankona-core, 100.97.248.107):

| Daemon | Role | Stations Served |

|--------|------|-----------------|

| `forge` | Station operator — batch attribute processing, listing copy generation, localization, ontology mapping, research extraction | 1A, 1B, 2 |

| `atlas` | Intelligence synthesis — market analysis, competitive tracking, buyer signals, boardroom reports, research subway routing | 3, 4, 5, 6, 7 |

| `sensei` | Coherence gate — quality control across all stations, session oversight, plan review | All (cross-cutting) |

Edge (rankona-edge, VPS):

| Daemon | Role | Stations Served |

|--------|------|-----------------|

| `watcher` | Compliance sentinel — TOS monitoring across 11 geographies, regulatory change detection, system health, alert pipeline | Compliance Sentinel |

| `sandman` | Operations — backup coordination, queue health, deploy verification, system metrics | 8 (Performance Reporting) |

| `delphi` | Query & retrieval — Telegram bot backend (@RankonaBot), semantic search routing, briefing assembly, client-facing query handling | All (query interface) |

**Growth path**: Add `avery` (client pipeline management) and `sterling` (profit-share tracking) when JV channel activates. Add second `forge-worker` on core if batch processing saturates.

Phase 2 — Neo4j Indexes & Constraints

2.1 — Full-Text Indexes (BM25 — powers mig_search)

```cypher

-- Memory full-text index

CREATE FULLTEXT INDEX memory_fulltext IF NOT EXISTS

FOR (m:Memory)

ON EACH [m.content, m.category, m.domain]

OPTIONS {indexConfig: {`fulltext.analyzer`: 'standard-no-stop-words'}};

-- ResearchMemory full-text index

CREATE FULLTEXT INDEX research_memory_fulltext IF NOT EXISTS

FOR (r:ResearchMemory)

ON EACH [r.content, r.domain, r.source_title]

OPTIONS {indexConfig: {`fulltext.analyzer`: 'standard-no-stop-words'}};

-- ResearchDocument full-text index

CREATE FULLTEXT INDEX research_document_fulltext IF NOT EXISTS

FOR (d:ResearchDocument)

ON EACH [d.title, d.summary, d.domain]

OPTIONS {indexConfig: {`fulltext.analyzer`: 'standard-no-stop-words'}};

-- Claim full-text index

CREATE FULLTEXT INDEX claim_fulltext IF NOT EXISTS

FOR (c:Claim)

ON EACH [c.content, c.domain]

OPTIONS {indexConfig: {`fulltext.analyzer`: 'standard-no-stop-words'}};

-- Product full-text index (Amazon-specific)

CREATE FULLTEXT INDEX product_fulltext IF NOT EXISTS

FOR (p:Product)

ON EACH [p.title, p.asin, p.sku]
```

Phase 5 — Daemon Infrastructure

5.1 — Daemon Wrapper

Create `/opt/rankona/scripts/daemon-wrapper.sh` on both nodes:

```bash

#!/usr/bin/env bash

set -euo pipefail

DAEMON_NAME="${1:?Usage: daemon-wrapper.sh <daemon-name>}"

Source environment

source /etc/rankona/env

Allowlist gate

ALLOW_FILE="/etc/rankona/daemons.allow"

if [[ -f "$ALLOW_FILE" ]] && ! grep -qx "$DAEMON_NAME" "$ALLOW_FILE"; then

echo "BLOCKED: $DAEMON_NAME not in $ALLOW_FILE" >&2

exit 1

fi

echo "[$(date -Iseconds)] Starting $DAEMON_NAME"

Hand off to Python daemon runner

exec /opt/rankona/.venv/bin/python \

/opt/rankona/scripts/run_daemon.py "$DAEMON_NAME"

```

Appendix B — 2-Node Failure Modes

| Scenario | Impact | Recovery |

|----------|--------|----------|

| Core down (power at Bitler) | Heavy ops unavailable, edge keeps running, Telegram responds "engine offline" | Restore power, Tailscale auto-reconnects, NATS drains queued events |

| Edge down (VPS outage) | Public access lost, core keeps running locally, compliance sentinel offline | Restore VPS (~15 min), zero data loss |

| Tailscale down between nodes | Edge serves static content, core operates locally | Tailscale auto-heals (minutes), manual fallback: direct IP if needed |

| Both down | Full outage | Offsite restore, 2-4 hours depending on backup size |
