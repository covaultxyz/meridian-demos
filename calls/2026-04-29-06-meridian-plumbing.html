Summary

A technical runbook detailing 13 phases for wiring the MERIDIAN intelligence stack into the RPM Capital ghost terminal, covering environment setup through full verification.

Decisions Made

  • Use local Ollama with nomic-embed-text for embeddings instead of a remote embedding service.
  • Use Claude CLI via OAuth for LLM access rather than API keys.
  • NATS on the RPM terminal is isolated from the main MERIDIAN mesh.
  • Neo4j GDS plugin is required for PageRank and community detection analytics.
  • Gardener runs daily at 03:00 UTC and backups run at 04:00 UTC.

Key Discoveries

  • The write validator is built into mig_remember() via the MCP server and requires no separate setup beyond having ontology.yaml in place.
  • Session coordination enables file conflict detection between Claude Code sessions, not just daemon-to-daemon messaging.
  • Without Neo4j indexes, writes bypass validation entirely — not just slower, but unvalidated.

Phase 2 — Neo4j Indexes & Constraints

After MIG seed import (runbook 05), create the indexes that power search, analytics, and write validation.

2.1 — Full-Text Indexes (BM25 — powers mig_search)

```cypher

-- Memory full-text index

CREATE FULLTEXT INDEX memory_fulltext IF NOT EXISTS

FOR (m:Memory)

ON EACH [m.content, m.category, m.domain]

OPTIONS {indexConfig: {`fulltext.analyzer`: 'standard-no-stop-words'}};

-- ResearchMemory full-text index

CREATE FULLTEXT INDEX research_memory_fulltext IF NOT EXISTS

FOR (r:ResearchMemory)

ON EACH [r.content, r.domain, r.source_title]

OPTIONS {indexConfig: {`fulltext.analyzer`: 'standard-no-stop-words'}};

-- ResearchDocument full-text index

CREATE FULLTEXT INDEX research_document_fulltext IF NOT EXISTS

FOR (d:ResearchDocument)

ON EACH [d.title, d.summary, d.domain]

OPTIONS {indexConfig: {`fulltext.analyzer`: 'standard-no-stop-words'}};

-- Claim full-text index

CREATE FULLTEXT INDEX claim_fulltext IF NOT EXISTS

FOR (c:Claim)

ON EACH [c.content, c.domain]

OPTIONS {indexConfig: {`fulltext.analyzer`: 'standard-no-stop-words'}};

```

2.2 — Vector Indexes (powers mig_recall semantic re-ranking, mig_hybrid_search)

```cypher

-- Memory vector index (768-dim for nomic-embed-text)

CREATE VECTOR INDEX memory_embedding IF NOT EXISTS

FOR (m:Memory) ON (m.embedding)

OPTIONS {indexConfig: {

`vector.dimensions`: 768,

`vector.similarity_function`: 'cosine'

}};

-- ResearchMemory vector index

CREATE VECTOR INDEX research_memory_embedding IF NOT EXISTS

FOR (r:ResearchMemory) ON (r.embedding)

OPTIONS {indexConfig: {

`vector.dimensions`: 768,

`vector.similarity_function`: 'cosine'

}};

```


Phase 5 — Daemon Infrastructure

5.1 — Daemon Wrapper

Create `/opt/rpm-capital/scripts/daemon-wrapper.sh` — the outer shell that every systemd unit calls. Provides:

• Node allowlist gate (prevents wrong daemons from starting)

• Exponential backoff on repeated failures

• Environment sourcing

```bash

#!/usr/bin/env bash

set -euo pipefail

DAEMON_NAME="${1:?Usage: daemon-wrapper.sh <daemon-name>}"

Source environment

source /etc/rpm-capital/env

Allowlist gate

ALLOW_FILE="/etc/rpm-capital/daemons.allow"

if [[ -f "$ALLOW_FILE" ]] && ! grep -qx "$DAEMON_NAME" "$ALLOW_FILE"; then

echo "BLOCKED: $DAEMON_NAME not in $ALLOW_FILE" >&2

exit 1

fi

echo "[$(date -Iseconds)] Starting $DAEMON_NAME"

Hand off to Python daemon runner

exec /opt/rpm-capital/.venv/bin/python \

/opt/rpm-capital/scripts/run_daemon.py "$DAEMON_NAME"

```

5.3 — Write Validator

The write validator (`brain/write_validator.py`) prevents junk writes to the graph. It needs:

• Category validation (fact, decision, preference, observation, learning)

• Binary content detection (rejects base64, binary blobs)

• Importance soft-demote for unprivileged categories

• Domain validation against ontology.yaml

This is part of the meridian codebase and works automatically when the MCP server is running — it's called internally by `mig_remember()`. No separate setup needed beyond having `config/ontology.yaml` in place.

Verify:

```bash
From Claude Code: mig_remember("Test memory", category="observation", importance=5)

This should be rejected (binary content)
```

Phase 6 — Session Coordination

Session coordination enables multi-daemon awareness — daemons and Claude Code sessions can see each other, send messages, hand off work, detect file conflicts, and gate each other.

6.1 — Session Register at Boot

Every daemon and Claude Code session calls `session_register()` at boot. This is handled automatically by:

• The MCP server (for Claude Code sessions) — `session_register()` tool

• The daemon runner (for systemd daemons) — called in the boot protocol

The session register publishes to `CLAUDE_SESSIONS` KV bucket (created in Phase 3).

6.2 — Boot Protocol for Daemon Contexts

Each daemon's system prompt (in its `.md` context file) should include this boot protocol:

```

Boot Protocol

1. session_register(task_description="<daemon role>")

• If had_compaction: true → call /resume-session

2. mig_recall(category="calibration", importance_min=7, limit=3)

3. session_inbox() — catch operator/peer messages

4. Declare ready in ≤30 words

```

6.3 — Session Tools Available After Phase 4

Once the MCP server is running (Phase 4), these session tools are automatically available:

| Tool | Purpose |

|------|---------|

| `session_register()` | Register with mesh, enable visibility |

| `session_inbox()` | Read messages from other sessions |

| `session_send(to, message)` | Send message to another session/daemon |

| `session_broadcast(message)` | Broadcast to all active sessions |

| `session_handoff(description, next_steps)` | Create handoff for another session |

| `session_accept_handoff(from)` | Accept a handoff |

| `session_check_conflict(file_path)` | Check if another session is editing a file |

| `session_gate(daemon, action, message)` | Halt/redirect/clear another daemon (supervision) |

| `session_map()` | Operator view of all sessions + progress |

| `session_report(accomplishment, blocked, action)` | Report progress |

6.4 — Gate Hooks (Optional but Recommended)

Gate hooks allow Sensei to halt other daemons mid-work. On MERIDIAN, this is a Claude Code PreToolUse hook. If RPM daemons run as Claude Code sessions:

Phase 10 — Gardener (Memory Maintenance)

The Gardener keeps the MIG healthy over time:

• **Decay enforcement** — reduce importance scores on aging memories per importance_rubric.yaml half-lives

• **Orphan cleanup** — remove disconnected nodes with no relationships

• **Concept deduplication** — merge duplicate concept nodes

• **Soft-delete enforcement** — ensure archived memories stay archived

10.1 — Gardener as Scheduled Job

```bash

Run gardener daily at 03:00 UTC

cat <<EOF > /etc/systemd/system/rpm-gardener.timer

[Unit]
```

Phase 12 — Prompt Enrichment & Context

12.1 — Daemon Context Files

Each daemon needs a `.md` context file that defines its identity, capabilities, boot protocol, and operating heuristics. These go in `/opt/rpm-capital/config/daemon-contexts/`:

| Daemon | Context File | Key Sections |

|--------|-------------|-------------|

| LAS | `las.md` | Write orchestration rules, schema, ingest patterns |

| Sensei | `sensei.md` | Coherence gate, session oversight, boot protocol |

| Delphi | `delphi.md` | Retrieval patterns, dossier assembly, search strategies |

| Watcher | `watcher.md` | Health check intervals, alert thresholds, escalation rules |

| Sandman | `sandman.md` | Backup schedules, ops oversight, queue health |

| Atlas | `atlas.md` | Intelligence synthesis, knowledge fitness, briefing generation |

| Avery | `avery.md` | Deal pipeline, TwentyCRM integration, scoring rubrics |

| Sterling | `sterling.md` | Fund economics, waterfall structures, capital analysis |

| Martin | `martin.md` | Legal cortex activation, compliance workflows, contract review |

Each context file should include:

• Identity and role description

• Boot protocol (session_register → mig_recall → session_inbox → declare ready)

• Authority boundaries (what the daemon may/must not do)

• MIG query patterns (what to search for, what categories to write)

• Proactive work loop (task pickup → execution → completion → next task)

• Phase checkpoint convention

• Calibration tags

12.2 — Prompt Enrichment

The brain/prompt_enricher.py module injects relevant memories into daemon prompts at task execution time. This works automatically through the MCP server and daemon runner — memories matching the task context are retrieved and injected.

Configuration in `settings.yaml`:

```yaml

retrieval:

reranking_enabled: true

mmr_enabled: true

mmr_lambda: 0.7

```

12.3 — Plan Recitation

For long-running sessions, plan recitation combats goal drift:

```

plan_recite(objectives="Complete due diligence on Deal X",

current_step="Step 3: Financial analysis")

plan_clear() # when done

```
