PHASE 4: BLIND SPOT ANALYSIS

Blind Spot 1: The Agreement Is Written From a Legal Template Perspective, Not a Technical Reality Perspective

**What is not being seen**: The agreement treats IP as a clean binary — it's either Company IP or Pre-Existing IP. But in practice, software IP exists on a spectrum. A configuration file that adapts a general-purpose framework to MBM's specific clinical workflow contains both: the framework (Pre-Existing) and the configuration (Company IP). The agreement has no mechanism for handling mixed-IP components — and in a platform-on-platform architecture, *most* components will be mixed.

**Why the author's position makes it invisible**: The drafting attorney works in legal abstractions (ownership, assignment, license). They don't see that line 47 of a Python file might import a general-purpose library (Pre-Existing), apply it to MBM patient data (Company IP), using a pattern the Contractor developed for a previous client (General Knowledge). The attorney sees "file = IP," not "file = layers of IP."

**Damage if unaddressed**: Every IP classification dispute will be expensive because the agreement forces a binary choice on inherently non-binary components. The proposed Section 3.6 (Layered Architecture Acknowledgment) partially addresses this, but even that treats layers as distinct when real code is interleaved.

Blind Spot 2: The Agreement Assumes MBM Can Function Without the Contractor

**What is not being seen**: The agreement gives MBM absolute, perpetual, worldwide ownership of all Company IP and a perpetual license to embedded tools. This implies MBM can independently operate, maintain, and evolve the software. But Omar is a psychiatrist with ~1 week of AI research (as of March 10). MBM has no engineering team. The IP assignment without technical capability to use it creates an asset MBM owns on paper but cannot operate in practice.

**Why the author's position makes it invisible**: From a legal perspective, ownership = control. The attorney assumes that owning IP means having the ability to use it. In software, this is false. Owning source code without the expertise to build, deploy, debug, and maintain it is like owning a factory with no one who knows how to run the machines.

**Damage if unaddressed**: If the relationship ends (for any reason), MBM has legal rights to software it cannot maintain. This either forces a fire-sale engagement with a new developer (who must learn the entire codebase), or the software degrades. This is also a business continuity risk for Omar's patients. The escrow recommendation in Exhibit D is the right instinct but it's non-binding.

Blind Spot 3: The Act 60 Compliance Framework Assumes Both Parties' Decrees Will Hold

**What is not being seen**: The IRS has intensified enforcement against Act 60 participants (Exhibit D acknowledges this). The agreement structures everything around the assumption that both MBM's and Contractor's Act 60 status remains intact. But §7.3(b) reveals MBM is still *in the process of amending* its decree to include SaaS/AI/IP — the amendment hasn't been approved yet. If the amendment is denied, MBM's Act 60 decree covers "Centralized Management Services and Telemedicine" — not software development.

**Why the author's position makes it invisible**: The attorney drafted the Act 60 sections to protect the existing structure. But the existing structure assumes a future event (decree amendment approval) that hasn't occurred. The attorney can see the current compliance obligations but not the conditional dependency: if the amendment fails, the entire compensation structure may need restructuring.

**Damage if unaddressed**: If MBM's decree amendment is denied, payments to Contractor for software development may not qualify as export services income. This could trigger IRS reclassification, retroactive tax liability, and potentially jeopardize both parties' Act 60 status.

Blind Spot 4: The Agreement Optimizes for IP Protection at the Cost of Operational Agility

**What is not being seen**: Every IP-protective mechanism (POA, ROFR, subcontractor approval, ethical walls, segregation, conflict disclosure) adds friction to the working relationship. For an early-stage engagement between a solo practitioner (Omar) and a solo contractor (Alex) working iteratively on evolving requirements, this level of IP apparatus will slow development and create administrative overhead disproportionate to the actual IP risk.

**Why the author's position makes it invisible**: The attorney's job is to protect against worst-case scenarios. They can't see that enforcing all provisions simultaneously — separate repos + separate credentials + conflict disclosure + subcontractor approval + information barriers + ethical walls — on a 2-person engagement is like putting a Boeing 747 cockpit in a Cessna.

**Damage if unaddressed**: Not legal damage, but operational damage. The agreement creates obligations that neither party will actually follow (because the overhead is impractical for the team size), which means both parties are technically in breach from day one. Selective enforcement then becomes a leverage tool.

Blind Spot 5: The Ayala PC Payment Routing Is an IRS Red Flag

**What is not being seen**: The Recitals state that Ayala PC (NJ entity) advanced payments to Contractor "on MBM's behalf as administrative convenience." From an IRS perspective, this pattern — US mainland entity pays PR-based contractor, PR entity claims the IP ownership — is a textbook arrangement that triggers Act 60 audit scrutiny. The "administrative convenience" framing doesn't change the economic substance: a NJ entity paid for services that generated IP claimed by a PR entity.

**Why the author's position makes it invisible**: The attorney focused on documenting the *intent* (MBM was always the real party).
