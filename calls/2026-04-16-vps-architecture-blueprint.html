Summary

Architecture blueprint for a 2-node Rankona cluster with an 8GB VPS as the public-facing edge node and a dedicated machine at 100.97.248.107 as the compute core.

Decisions Made

  • Split responsibilities into core/edge: VPS handles public-facing and always-on services, dedicated machine handles stateful and compute-heavy workloads.
  • NATS hub lives on the VPS edge node so coordination survives core outages.
  • Compliance sentinel runs on VPS to guarantee 24/7 uptime for time-sensitive TOS and regulatory alerts.
  • Zero open inbound ports on either node; all public ingress routes through Cloudflare Tunnel.
  • VPS-to-core communication uses Tailscale private tunnel only; SSH from VPS to core is disabled.

LENS — Why Two Nodes

The dedicated machine at Bitler (100.97.248.107) is the **engine room** — heavy compute, large Neo4j, embedding service, batch workers. It sits behind UPS and redundant internet, but it's still a physical machine on a business-premises connection. Power blip, ISP outage, router reboot — any of these interrupts the terminal.

The VPS is the **concourse** — the public station where passengers enter. It doesn't do the heavy work. It's reachable, reliable, and always-on. When Carl's team sends a Telegram message at 3am during a Stockholm power outage, the VPS responds. When Amazon publishes a TOS change at midnight local time, the compliance sentinel on the VPS catches it.

This is a classic **core/edge split**. The dedicated machine is the core (stateful, compute-heavy, infrequently reached). The VPS is the edge (always-on, public-facing, lightweight).

Traffic Flow

• **Inbound from internet** → Cloudflare Tunnel → VPS (zero open ports on either node)

• **VPS ↔ Core** → Tailscale private tunnel (Neo4j queries, heavy ops requests)

• **Core ↔ External services** (Amazon API, scrapers) → via VPS as egress proxy (optional, for IP consistency) or direct via Cloudflare WARP

---

VPS SERVICE STACK

Always-On Services (baseline ~2GB RAM)

**1. NATS Server** (~200MB)

• Central coordination hub for the 2-node cluster

• Core daemon on rankona-core connects here as a client

• Survives rankona-core outages — queues messages until core returns

• Single shared `RANKONA_NATS_AUTH_TOKEN` (per ghost terminal isolation)

• JetStream enabled for durable message streams (appeal outcomes, TOS changes)

**2. Telegram Bot** (~100MB)

• `@RankonaBot` — Carl's team query interface

• Long-polling or webhook mode (webhook via Cloudflare Tunnel preferred)

• Routes queries to rankona-core via NATS request/reply

• If core is offline: graceful degradation ("engine offline, try again in 5 min")

**3. Compliance Sentinel** (~200MB)

• Continuous monitor for Amazon TOS changes across 11 geographies

• Government regulatory change detection per country per category

• Publishes findings to NATS (`compliance.signal`) — core daemon ingests when online

• Works independently of rankona-core — dedicated sentinel loop

• Critical: this is WHY it belongs on VPS — needs 24/7 uptime for time-sensitive alerts

**4. Cloudflare Tunnel** (~100MB)

• Multiple ingresses: telegram webhook, dashboard, reports, vox endpoint, API gateway

• Outbound-only — zero open ports

• Managed DNS via Cloudflare (SOC2-compliant path)

**5. Caddy** (~200MB)

• Reverse proxy + static site host

• Auto-TLS via Let's Encrypt (through Cloudflare DNS-01)

• Serves: `dashboard.rankona.speedoftrust.ai`, `reports.rankona.speedoftrust.ai`, per-client subdomains

On-Demand Services (peak ~2GB additional)

**6. Report/Website Generator** (~500MB peak)

• Ronin worker pattern — generates HTML/PDF from MIG queries

• Triggered by NATS cron events or manual request via Telegram

• Pulls data from rankona-core Neo4j over Tailscale

• Writes to Caddy static dir or pushes to Cloudflare Pages

**7. Vox Agent Endpoint** (~500MB peak)

• Voice message handler for Telegram

• STT (speech-to-text) for incoming voice queries → routes to core

• TTS (text-to-speech) for outgoing voice briefings

• Can call Gemini/Groq for STT+TTS if Claude CLI OAuth rate-limited

**8. restic Backup Receiver** (~100MB)

• Receives encrypted backups from rankona-core every 6 hours

• Forwards to offsite storage (S3-compatible, e.g. Backblaze B2)

• Retention: 7 daily, 4 weekly, 12 monthly

Reserved (safety margin ~2GB)

• OS, kernel, syslog, Tailscale daemon

• systemd, Docker daemon (if used)

• Caddy buffers during traffic spikes

• Headroom for vox/report generator concurrency

**Total expected footprint**: 4-5GB under normal load, 6GB peak. 8GB VPS has adequate headroom.

FAILURE MODES & RESILIENCE

VPS goes down

• **Public access lost**: Telegram, dashboards, reports all unreachable

• **Core keeps working**: local batch jobs continue, ingestion pipeline runs

• **NATS queue backs up**: messages buffer, replay when VPS returns

• **Recovery**: restore VPS, Tailscale reconnects automatically, NATS resumes

• **SLA impact**: ~15 min for VPS recovery, zero data loss

Core goes down (power outage at Bitler)

• **Heavy ops unavailable**: no new MIG queries, no appeals, no attribute processing

• **VPS stays up**: Telegram bot responds "engine offline" gracefully

• **Compliance sentinel keeps running**: still watches TOS, queues findings

• **Backup replication pauses**: last successful backup still available

• **Recovery**: restore core, Tailscale reconnects, NATS drains queued events, backfill from compliance sentinel buffer

Both nodes down (rare)

• Offsite backup restore procedure

• New VPS + new dedicated machine provisioned from runbook

• MIG restored from most recent offsite snapshot

• Expected downtime: 2-4 hours depending on backup size

Tailscale down between nodes

• VPS serves static content, caches, queued messages from NATS

• Core operates locally but can't receive new work

• Tailscale auto-heals within minutes typically

• If sustained: fallback to Cloudflare Tunnel-to-Cloudflare Tunnel mesh (future)

DEPLOYMENT STRATEGY

VPS Provisioning Sequence

1. **Base hardening** (following SOC2 baseline)

• Disk encryption enabled

• SSH key-only auth, fail2ban, unattended-upgrades

• UFW: deny all inbound except Tailscale + Cloudflare Tunnel outbound

2. **Tailscale join**

• Separate tailnet from MERIDIAN core mesh (ghost terminal isolation)

• `rankona-tailnet` with ACLs limiting VPS ↔ core communication only

3. **NATS server**

• Single shared token in `/etc/rankona/secrets.env`

• JetStream enabled, storage on local disk

• systemd unit with restart-on-failure

4. **Cloudflare Tunnel**

• Tunnel credentials in secrets

• Ingress rules: telegram, dashboard, reports, vox, api

• DNS managed via Cloudflare (zone: rankona.speedoftrust.ai or similar)

5. **Caddy**

• Site config per subdomain

• Auto-TLS via DNS-01 challenge

6. **Application services** (Telegram bot, compliance sentinel, report generator, vox agent)

• Each as systemd unit

• Logs to journald, backed up via journal-to-file daily

7. **Backup receiver**

• restic repository setup

• SSH key auth from rankona-core to VPS

• Offsite forwarder configured (B2 or similar)

Connection to Core

• Tailscale key exchange between nodes

• NATS: core configured to connect to VPS:4222 as hub (not the reverse)

• Neo4j remote access from VPS: allowed via Tailscale only, bolt+ssc TLS

• SSH from VPS to core: disabled (security boundary)

• SSH from core to VPS: allowed for backup push only

COST/CAPACITY NOTES

• 8GB RAM is adequate for the edge workload described (4-5GB baseline, 6GB peak)

• If vox agent + report generation are concurrent and heavy: brief spikes possible → monitor

• If future load grows: scale up VPS before adding second edge node

• Disk: ~20GB should be sufficient (NATS JetStream, Caddy logs, backup spool, static sites)

• Bandwidth: dominated by backup replication + report serving — should fit in typical VPS allotments
