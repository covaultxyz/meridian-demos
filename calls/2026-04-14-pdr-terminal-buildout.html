Data Contracts Between Stations

Every station consumes input and produces output. Without defined contracts, integration breaks at the seams.

| Interface | Producer | Consumer | Format | Contract |

|-----------|----------|----------|--------|----------|

| Category report → Ontology | Jennifer (manual export) | 1A-2 parser | CSV/TSV flat file from Seller Central | Columns: ASIN, category, all attributes; encoding: UTF-8; size: up to 43K rows |

| Ontology → Attribute inference | 1A-1 mapper | 1A-4 engine | Neo4j nodes: `(:Category)-[:HAS_ATTRIBUTE]->(:Attribute)` | Each Attribute has `name`, `type`, `required`, `cosmo_priority` |

| Attribute inference → Bulk upload | 1A-4 engine | 1A-6 uploader | JSON: `{asin, marketplace, attributes: [{name, value, confidence, source}]}` | Confidence ≥0.8 auto-uploads; <0.8 flagged for human review |

| Bulk upload → Rejection diagnostics | Amazon API | 1A-8 parser | XML error response (SP-API format) | Parsed to `{asin, field, error_code, suggested_fix}` |

| Keyword optimizer → Copy generator | 1B-1 optimizer | 1B-2 generator | JSON: `{asin, primary_kw, secondary_kw[], tertiary_kw[], embedded_phrases[]}` | Max 250 chars backend; primary in title; embedded phrases cover 2+ targets |

| Copy generator → Approval workflow | 1B-2 generator | 1B-5 workflow | Structured listing: `{title, bullets[5], description, backend_keywords}` | Round-trip: generate → review → revise → approve; ≥90% first-round target |

| Sentinel → Alert pipeline | CS-1/CS-2 | CS-4 Telegram | JSON: `{change_type, geography, category, impact_asins[], severity}` | Severity: critical (suspension risk), high (listing suppression), medium (ranking impact), low (informational) |

LLM Cost & Throughput Model

Claude CLI is rate-limited at ~45 calls per 5-hour window. This constrains batch extraction.

| Operation | LLM Provider | Calls per Batch | Est. Cost/Batch | Notes |

|-----------|-------------|-----------------|-----------------|-------|

Throughput constraint:** At 43K SKUs, attribute inference alone requires ~8,600 LLM calls. At Claude CLI rates, that's ~956 five-hour windows (~199 days). With Groq offload at ~100 calls/min, same batch completes in ~86 minutes. **Groq offload is not optional — it's architecturally required for batch operations.

**Monthly LLM cost estimate (steady state):** $50-150/month at current volume. Scales linearly with SKU count and market count. Fjällräven (global, many markets) could 3-5x this.

Perspective 4: Reusability (Platform)

The terminal isn't just for Rankona. Carl's JV channel targets $100M+/year sellers at $10K-15K/month. Every component must be built with multi-instance deployment in mind.

What must be instance-specific:

• Neo4j database (already scoped via `--instance-id`)

• NATS namespace (already scoped)

• Telegram bot token (per-client)

• Cloudflare Tunnel (per-client subdomain)

• Category ontology mappings (per-client, per-category)

• Knowledge seed data (per-client)

What should be shared across instances:

• Embedding model (all-mpnet-base-v2 — same model, different data)

• Claude CLI (OAuth — shared auth, isolated prompts)

• Station code (same codebase, different config)

• Dashboard framework (same UI, different data source)

• Appeal generator (same templates, different case history)

**Design constraint:** Every station must accept `instance_id` as a configuration parameter. No hardcoded paths, no shared databases, no cross-instance data leakage. The `customer-deploy.sh --instance-id` pattern already enforces this at Layer 0. Stations must honor it.

Perspective 5: What Could Go Wrong (Adversarial)

1. **Ontology drift.** Category mappings that work for horse equipment may not transfer to Fjallraven's outdoor gear. Each new category is a fresh mapping exercise ($2,500 each). If the mapping process isn't genuinely reusable, the "done once per category" promise becomes "done once per client per category."

2. **Knowledge leakage.** Carl wants exclusivity ("tailor-made"). If the codebase is the same across instances, what's actually exclusive? The answer must be: the ontology mappings, the category knowledge, the accumulated intelligence — not the code. KaaS monetizes process knowledge, not platform code.

3. **Exit valuation dependency.** 50/50 above $25M means Meridian's upside is tied to an acquisition that may never happen. The monthly retainer ($4,500) and profit share (20%) are the load-bearing revenue. Exit participation is optionality, not income.

4. **Scope expansion before proof.** Jennifer's wish list (PPC, account health, image QA) expands the scope beyond 8 stations. The $4,500/month covers "less than a strategist" — if scope grows, the unit economics break.
