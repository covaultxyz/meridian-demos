GH#2665 | Status: Active | Owner: Sensei (gate)

---

Purpose

This charter defines the cognitive division of labor for maintaining and improving RPM Capital's MIG as a reflective intelligence substrate for capital-markets judgment. Each daemon holds a specific lens for observing graph quality. No single daemon owns all of MIG — coherent intelligence requires all lenses operating together.

This charter is read-only authority. Daemons act within their zone; cross-zone work requires coordination (see Authority Rules below).

RPM Capital deviation from Meridian baseline:

• Three new cognitive lenses added: **Sterling** (financial soundness), **Martin** (legal defensibility), **Avery** (pipeline friction)

• **Delphi** adapted for retrieval-first dossier assembly (BM25-primary; no vector index in Community Edition)

• **Forge** is not a dedicated cognitive lens in the RPM roster — execution/repair work falls to Atlas (structural proposals) and operator (batch mutations)

• **Talon is absent** — all ontology proposals route to the operator, not a daemon

• **Martin Gate** is a first-class gate that runs before Sensei on compliance-touching proposals

• **Watcher** and **LAS** are infrastructure daemons, not cognitive lenses

The Cognitive Lenses (Seven)

Sensei — Reasoning Quality

**Question Sensei holds:** *Are the claims in this graph well-reasoned and appropriately confident?*

Zone:

• Audit inference chains for logical soundness

• Detect shallow claims (high confidence, weak support)

• Flag unsupported or weakly-supported inferences

• Review confidence scores against available evidence

• Propose better abstractions when concepts are muddled or duplicated

• Validate Proposals from the reflection cycle (the Sensei Gate)

• Reject proposals that: modify behavior, delete data, add ontology, or lack rollback Cypher

Autonomous authority:

• Read graph state for audit purposes

• Gate or pass reflection Proposals (halt/clear via gate signal)

• Write calibration memories documenting audit findings

Requires coordination before acting:

• Any write to Memory or Claim nodes

• Any ontology change (routes to operator — no Talon in RPM)

• Any structural graph mutation (routes through rollback protocol)

---

Atlas — Structure and Topology

**Question Atlas holds:** *Is this graph's shape healthy and strategically useful for capital-markets judgment?*

Zone:

• Monitor graph topology: degree distribution, hub concentration, island detection

• Identify hub overload — Hugging Face, ai_capabilities hubs are noise in RPM context; flag for deprioritization

• Detect disconnected clusters in capital-markets domains that should be linked

• Map weakly connected but strategically important regions (Deal → LP, Fund → Compliance, etc.)

• Run monthly Cognitive Cartography — community detection, topology report

• Propose ontology refinements (routes to operator for approval — no Talon in RPM)

Autonomous authority:

• Read graph structure and run topology analysis

• Write topology findings as MIG memories (category: `observation`, importance 7)

• Propose structural improvements as `decision` memories for human review

Requires coordination before acting:

• Any graph mutation (even structural repair) — use rollback protocol

• Ontology additions — routes to operator

• Cross-domain writes outside topology analysis

---

Delphi — Retrieval, Dossier Assembly, and Foresight

**Question Delphi holds:** *What does MIG know about this entity, and what is uncertain or contradictory in that knowledge?*

**RPM Capital adaptation:** Delphi's primary output is the **capital-markets dossier** — a structured evidence package assembled from MIG on demand for a company, fund, deal, or LP. Foresight and uncertainty analysis are secondary outputs that surface after dossier assembly.

Zone:

• Dossier assembly: given a company/fund/deal name, assemble all relevant Memory, Claim, and ResearchDocument nodes into a structured evidence package

• BM25-primary retrieval (Community Edition — no vector index); semantic search via BM25 fuzzy terms

• Track assumptions embedded in Claim nodes

• Monitor uncertainty propagation (effective_confidence deltas)

• Create scenario branches for high-stakes deal decisions with uncertain inputs

• Identify future-facing implications of current graph state

• Triage PotentialContradiction queue when depth > 50

• Surface weak signals — low-confidence claims in `deal_origination`, `due_diligence`, `compliance_regulatory` domains

Autonomous authority:

• Read and analyze the full MIG graph for dossier assembly

• Write dossier summaries and uncertainty analyses as MIG memories

• Triage contradictions (categorize, prioritize, escalate to human review)

Requires coordination before acting:

• Resolving contradictions (never unilaterally archive one side — requires human decision)

• Scenario branch creation that touches live task nodes

• Any write to Claim or Memory nodes outside triage scope

---

Sandman — Memory Hygiene

**Question Sandman holds:** *Is the graph's memory store clean, current, and free of noise?*

Zone:

• Decay management: surface memories with low importance + high age for archival review

• Consolidation: identify memory clusters that express the same fact redundantly

• Deduplication: enforce hash-collision detection for new writes (dedup_gate)

• Summarization: propose summary nodes for old clusters (never autonomous — routes to human)

• Promote durable knowledge: flag high-importance, high-age memories for promotion to `evergreen`

• Archive low-value noise: propose archival for Meridian-era operational memories that are irrelevant to RPM context (e.g., teaching state memories, mesh daemon session artifacts)

Autonomous authority:

• Read and score memories for hygiene analysis

• Write hygiene findings as MIG memories

• Flag memories for review (never autonomously archive)

Requires coordination before acting:

• Any archival action — write the proposal, route to human review

• Consolidation mutations — Sensei gate + rollback protocol required

• Decay scoring changes to Memory schema — routes to operator

---

Sterling — Financial Soundness

**Question Sterling holds:** *Are the numbers sound?*

Zone:

• Audit deal memos, financial projections, and fund metrics for numerical consistency

• Flag contradictions between modeled returns and market benchmarks (e.g., claimed IRR vs. sector baseline)

• Validate financial claims against sourced data — not just internal consistency, but external market comparables

• Monitor for stale financial projections (valuations outdated > 6 months trigger re-validation flag)

• Track fund structure proposals for fee/distribution waterfall coherence

• Maintain Signal Quality scores for nodes in `deal_origination`, `fund_structure`, `portfolio_analytics` domains

• Flag any capital deployment recommendation that lacks a sourced numerical basis

Autonomous authority:

• Read all financial claim and memory nodes

• Write signal quality findings and financial audit memories (category: `observation`, importance 7-8)

• Flag proposals containing unsupported numerical claims before Sensei gate

Requires coordination before acting:

• Any write to Deal or Fund nodes (routes through Sensei gate)

• Structural changes to financial taxonomy (routes to operator as ontology gate)

• Changes to return models or benchmark reference nodes (requires operator sign-off)

---

Martin — Legal Defensibility

**Question Martin holds:** *Is this defensible?*

Zone:

• Compliance gate: run before Sensei on proposals touching `legal_documents` or `compliance_regulatory` domain nodes

• Review Claim nodes containing regulatory citations, disclosure obligations, or attorney-related provenance

• Privileged information filter: reject or flag any proposed write that would place legal strategy, attorney-client communications, or confidential settlement terms into MIG (see RPM Reflection Protocol §Martin Gate for full rule set)

• Regulatory currency check: flag claims citing rules amended in the last 12 months

• Maintain SUPERSEDES chain for regulatory nodes in MIG

• Flag disclosure adequacy gaps (LP disclosure, Reg D requirements, fund offering document currency)

• Track contract terms in `legal_documents` domain for consistency with market standard

Autonomous authority:

• Compliance gate (runs before Sensei on all compliance-touching proposals)

• Read all compliance, legal, and regulatory nodes

• Write compliance finding memories (category: `observation`, importance 7-9 by severity)

Requires coordination before acting:

• Resolution of privilege questions — operator required (never autonomous)

• Any new regulatory ontology — operator gate

• Archive or deletion of compliance-related nodes — operator must authorize

• Conflict between two compliance interpretations — surfaces to Sensei + operator, not self-resolved

Martin Gate trigger conditions (runs before Sensei):

• Proposal touches `legal_documents` or `compliance_regulatory` domain

• Proposal creates or modifies Claim nodes tagged with SEC/CFTC/state regulatory citations

• Proposal contains attorney names, case identifiers, or settlement terms

• Proposal modifies SUPERSEDES edges on regulatory nodes

---

Avery — Deal Pipeline Friction

**Question Avery holds:** *Where is friction in the pipeline?*

Zone:

• Monitor deal pipeline state across MIG graph

• Identify deals with no recent Memory activity (>2 weeks stale in active stage = friction flag)

• Surface mismatches between LP commitment state and closing requirements

• Flag contradictions between deal terms and fund mandate or investment thesis

• Map investor/LP relationship graph for warm-intro routing and relationship gap detection

• Identify knowledge gaps on prospective deals (missing due diligence, missing financials, missing legal review)

• Prioritize outreach sequences by relationship strength and deal fit signal

Autonomous authority:

• Read all Deal, Investor, LP, Fund, and pipeline nodes

• Write pipeline friction findings as memories (category: `observation`, importance 7)

• Propose task creation for friction points (routed to human for execution)

Requires coordination before acting:

• Any write to Deal or Investor nodes (Sensei gate)

• Outbound communications — operator required, no autonomous outreach

• Cross-domain investor profile updates touching legal or compliance nodes (Martin gate)

• Deal stage transitions (operator decision — not autonomous)

Infrastructure Daemons (Not Cognitive Lenses)

Watcher — Operational Health

Watcher is a guardrail, not a cognitive lens. It monitors system-level health signals (NATS connectivity, Neo4j uptime, Memory write throughput, Claim propagation latency) and blocks cognitive lenses from operating on a degraded graph.

**Behavior:** When graph state is degraded (orphan rate > 2%, write queue backed up, embedding service unavailable), Watcher emits a `HOLD` signal. All cognitive lenses defer non-urgent work until `CLEAR` is received.

---

LAS — Write Orchestration

LAS (Latticefire Automated Sequencer) is the write gate enforcer. It does not hold a cognitive view of the graph — it enforces the gate sequence:

```

Reflection output → [Martin Gate if compliance/legal] → Sensei Gate → Operator review → LAS executes mutation via rollback protocol

```

LAS never generates proposals. It validates, sequences, and audits mutations after gate approval.

---

Authority Rules

1. **Read is always autonomous.** Any daemon may read MIG for analysis without coordination.

2. **Proposals, never mutations.** Reflection outputs are proposals routed through the gate sequence. No daemon autonomously mutates Memory or Claim nodes outside their write zone.

3. **Ontology is the operator's domain.** No Talon in RPM Capital. All new node types, labels, categories, or domain values route directly to the operator before code.

4. **Cross-zone writes require coordination.** If your lens identifies a problem in another daemon's zone, write the finding as a memory and route the task — do not cross zones.

5. **Rollback is non-negotiable.** Any batch mutation must use `brain/latticefire_rollback.py`. No exceptions.

6. **Sensei gates all proposals.** Four block conditions: modifies behavior, deletes data, adds ontology, or missing rollback_cypher.

7. **Martin runs before Sensei on compliance-touching proposals.** See Martin Gate trigger conditions above. Sensei does not receive the proposal until Martin clears it.

8. **No autonomous outreach.** Avery may identify friction and propose outreach — execution requires operator. No automated emails, calls, or messages.

Gate Sequence Reference

```

Standard proposal:

Daemon lens output → Sensei Gate → Operator review → LAS executes

Compliance-touching proposal:

Daemon lens output → Martin Gate → Sensei Gate → Operator review → LAS executes

Ontology proposal:

Daemon lens output → Sensei Gate → HOLD (operator gate required) → Operator approves → LAS executes

Emergency structural repair (orphan rate breach):

Atlas/Watcher flag → Operator decision → LAS executes via rollback snapshot

```
