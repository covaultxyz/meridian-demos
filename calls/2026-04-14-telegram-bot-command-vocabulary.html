Summary

The document outlines the comprehensive design and proposed features for @RankonaBot, a new Telegram bot aimed at enhancing Amazon operations with role-based commands, natural language queries, proactive alerting, and a mobile-first approach.

Decisions Made

  • The bot design prioritizes mobile-first, ensuring useful output within a single screen.
  • All bot responses will be role-scoped and filtered by the requesting user's workspace scope.
  • Natural language queries against the Amazon MIG will serve as a fallback if messages do not match a command pattern.
  • The bot will function as the delivery mechanism for CS-4 alert pipeline, sending unprompted notifications based on severity.
  • All commands are designed to be stateless, self-contained, and derive context from the MIG rather than chat history.
  • Natural language query execution will have a 10-second timeout.
  • Natural language query results will be capped at 20 rows.
  • No write operations will be allowed via natural language queries.
  • Critical alerts will be delivered immediately to owners and operations.
  • High-severity alerts will be delivered immediately during launch mode or in a daily digest during steady-state to owners and operations.
  • Medium-severity alerts will be included in a daily digest delivered at 07:00 local to owners and operations.
  • Low-severity alerts will be included in a weekly digest delivered on Monday at 07:00 to owners only.
  • Critical alerts unresolved after 24 hours will escalate directly to Carl.
  • Critical alerts unresolved after 72 hours will escalate to Delphi for investigation.
  • @RankonaBot will be a completely separate bot instance, without shared runtime code with OracleDelphiBot.
  • Workspace filtering will be applied at the query level for defense-in-depth, not at the response level.
  • Telegram message responses will adhere to a 4,096 character limit, splitting longer content.
  • Inline keyboards will be used for drill-down navigation in responses.
  • Markdown formatting will be used in bot responses.
  • Response time targets are less than 3 seconds for cached queries and less than 10 seconds for live MIG queries.
  • A rate limit of max 20 commands per user per minute will be enforced.

Commitments

  • Carl is committed to providing the bot token and SSH access, which are currently blocking the implementation of @RankonaBot.

Key Discoveries

  • The bot design incorporates a "defense in depth" strategy by applying workspace filtering directly at the query level, ensuring users only retrieve authorized data.
  • Groq primary, rather than Claude CLI, will be used for interpreting natural language queries to optimize resource usage.
  • @RankonaBot introduces advanced patterns such as role-based access control, workspace-scoped query filtering, inline keyboard navigation, proactive alert delivery, and multi-workspace switching, which were not present in OracleDelphiBot.

Design Principles

1. **Mobile-first.** Carl uses Telegram from his phone. Every command must return useful output within a single screen (~15 lines). Detailed data links to dashboard.

2. **Role-scoped.** Every response filters by the requesting user's role and workspace scope. Jennifer never sees Fjällräven data. Johan never sees non-localization stations.

3. **Natural language fallback.** If a message doesn't match a command, treat it as a natural language query against the Amazon MIG (like OracleDelphiBot's `/ask` pattern). Uses `brain/offload_llm.py` for interpretation.

4. **Alert delivery channel.** The bot is also the delivery mechanism for CS-4 alert pipeline — it sends unprompted notifications per severity routing.

5. **Stateless commands.** No multi-step wizards or conversation state. Every command is self-contained. Context comes from the MIG, not from chat history.

Natural Language Query (Fallback)

Any message that doesn't match a `/command` pattern is treated as a natural language query against the Amazon MIG.

Flow:

1. Extract intent via `brain/offload_llm.py` (Groq primary — never burn Claude CLI on bot queries).

2. Map intent to Cypher query template.

3. Execute against Neo4j (workspace-filtered by caller's role).

4. Format response.

Examples:

• "How are my saddles doing?" → conversion summary for horse equipment category

• "Any new competitor price changes?" → last 24h competitor alerts

• "What's the rejection rate this week?" → upload batch rejection aggregation

• "Show me the worst performing ASIN" → lowest conversion rate delta vs baseline

Guardrails:

• Query execution timeout: 10 seconds. If exceeded, respond with "Query too complex — try a specific /command instead."

• Results capped at 20 rows. Larger result sets return "Showing top 20. Use /dashboard for full data."

• No write operations via natural language. All MIG writes go through station pipelines.

Role-Based Filtering Implementation

Every command handler wraps its Neo4j query with workspace filtering:

```python

Pseudocode — actual implementation in telegram_rankona_bot.py

async def handle_command(update, context):

user = await get_bot_user(update.effective_user.id)

if not user:

await update.message.reply_text("Not authorized. Contact admin.")

return

Workspace filter applied to all queries

workspace_filter = user.workspace_ids # ['rankona'] or ['*'] for owner

Role-based command access check

if command not in ROLE_COMMANDS[user.role]:

await update.message.reply_text(f"/{command} not available for your role.")

return

Execute with workspace scope

results = await query_amazon_mig(cypher, workspace_ids=workspace_filter)

```

**Key implementation detail:** The workspace filter is applied at the query level, not the response level. The MIG never returns data the user shouldn't see — defense in depth.

Telegram Message Constraints

• **Max message length:** 4,096 characters (Telegram limit). Split longer responses.

• **Inline keyboards** for drill-down: e.g., `/alerts` shows summary with buttons [View Critical] [View High] [Resolve All].

• **Markdown formatting:** Bold for headers, monospace for ASINs/numbers, emoji for severity indicators.

• **Response time target:** <3 seconds for cached queries, <10 seconds for live MIG queries.

• **Rate limiting:** Max 20 commands per user per minute (prevents abuse, protects MIG).

Migration Path from OracleDelphiBot

@RankonaBot is a completely separate bot instance. No shared code with OracleDelphiBot at runtime. However, the implementation reuses patterns from `integrations/telegram_bot.py`:

| Pattern | Source | Adaptation |

|---------|--------|------------|

| Command handler registration | `telegram_bot.py:900-940` | Same `CommandHandler` pattern, new command set |

| Neo4j query wrapper | `telegram_bot.py` query functions | Point at Rankona instance (not prime:7750) |

| Rate limiting | `telegram_bot.py` rate limiter | Same approach, per-user limits |

| Markdown formatting | `telegram_bot.py` formatters | Reuse, add alert severity emoji |

| Natural language fallback | `telegram_bot.py` `/ask` handler | Same LLM pipeline, workspace-scoped |

New patterns (not in OracleDelphiBot):

• Role-based access control (RBAC) — per-user role lookup before every command

• Workspace-scoped query filtering — all queries carry `workspace_ids[]`

• Inline keyboard navigation — drill-down UX for mobile

• Proactive alert delivery — bot pushes to users, not just responds

• Multi-workspace switching — `/workspace` command for owner role
