Transduction, Not Recursion

The Sovereign Legal Transducer reorients the AI's operating mode from recursion to transduction. This is not a constraint on capability — it's a reorientation of how the AI processes your materials.

Recursion (Standard AI)
  • x Builds from internal patterns and predictions

How It Works

The attorney configures. The transducer locks. Output is constrained to source-attributable claims. Every step is auditable.

1

Configure (8 Questions)

The transducer asks 8 sequential questions: gap handling mode, required fields, generation permission, source materials, jurisdiction, output verbosity, PHI redaction, and request type. No defaults. No hidden parameters. Every choice is explicit.

2

Lock

After all questions are answered, the transducer locks. Configuration is frozen. The system restates your rules with your specific settings applied. No further changes mid-session. This prevents prompt drift.

3

Upload Source Materials

Upload contracts, depositions, emails, transcripts. Each document is chunked, hashed with SHA-256, and registered in a source hash registry. These hashes form the cryptographic bindings for verification.

4

Transduce

The AI produces output with inline provenance markers on every claim. The system prompt constrains the model to transduction mode — it transmits only what exists in your source materials.

5

Validate

Four-stage validation: marker presence check (every sentence), cryptographic hash verification (source binding), gap handling compliance (per your config), and hallucination scan (unsourced claims blocked).

6

Review Omissions

The omission detector highlights source content NOT referenced in the AI output. It does not judge significance — the attorney decides. This is assisted review, not automated fact detection.

7

Attest & Export

Create an attestation record: attorney signature, document hash, validation result, complete audit trail. Three signature methods: typed name, digital signature, or wet signature placeholder. SOC2-ready.

5-Layer Transduction Wrapper

The model does not operate directly. It operates inside a transduction wrapper that constrains, validates, and audits every output.

Layer 1 — Input

Source materials (PDF, DOCX, TXT, transcript) + user query + transducer configuration + source hash registry

Layer 2 — Prompt Construction

System prompt with transduction rules + full source injection + config injection + few-shot marker examples

Layer 3 — Model Inference

Claude API — temperature 0.0, top P 1.0, max 8192 tokens. Maximum determinism. No creativity.

Layer 4 — Output Validation

Marker presence check + source hash verification + gap detection + hallucination scan. Block if validation fails.

Layer 5 — Post-Processing

Source link resolution (hash match + semantic fallback) + omission highlighting + audit log generation + attestation preparation

Honest Claims Table

Every claim we make about this product comes with its limitation. The disclosure of limitations IS the product.

Feature Claim Limitation
Source Attribution Every factual claim carries an inline provenance marker Marker presence is enforced. Whether the marker points to the RIGHT source depends on model quality. Hash verification catches fabricated markers.
Hallucination Prevention Output is blocked if claims lack source binding The model still makes inferences internally — we cannot "block access to training data." We constrain the OUTPUT, not the inference process. Disclosure is prominent.
Omission Detection Unreferenced source content is highlighted for review The system highlights WHAT was omitted. It does NOT judge WHETHER the omission matters. The attorney makes that determination. This is assisted review, not automated analysis.
Cryptographic Binding SHA-256 hash binds each [T] marker to source chunk 8-character hash prefix for readability. Collision probability is negligible at this scale but not zero. Full hash available in audit log.
Zero-Inference We don't claim this. Zero-inference is impossible — the model always infers. We call it "inference-constrained": the model infers, the wrapper filters. What survives is source-attributable.
PHI Redaction Pattern-based redaction with [R] marker Pattern matching is not reliable for complete de-identification. For actual PHI, use Tier 1 (HIPAA BAA with Anthropic) or Tier 2 (self-hosted, no data leaves firm). DE-IDENTIFY mode was removed.
Audit Trail Complete session log with attestation In-memory session store for MVP. Production deployment requires PostgreSQL for durable session state and multi-instance support. Schema provided (audit_schema.sql).

Confidentiality Gate

Select your data processing tier before any source material is processed.

Tier 1: Production

HIPAA BAA in place. Data encrypted at rest and in transit. Anthropic API with signed BAA.

Attestation: "I have reviewed the BAA dated [date]"

Tier 2: Local Processing

No data leaves firm network. Requires self-hosted model (Llama 3.2 90B recommended).

Attestation: "I confirm local deployment"

Tier 3: Test / Synthetic

No real data. Synthetic documents only. For evaluation, training, and demos.

No attestation required

DE-IDENTIFY mode has been removed. Reliable de-identification requires human review. For PHI, use Tier 1 or Tier 2.
